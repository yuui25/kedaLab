# ML システムコンポーネントへの攻撃

> **このファイルの位置づけ：** ML ベースシステムの「System コンポーネント」（ハードウェア・OS・設定・デプロイ環境）を標的とした攻撃の原理と TTP をまとめる。`AI_Red_Teaming_Concepts.md` のコンポーネント分類における System 層の詳細ファイル。LLM 固有の DoS（Unbounded Consumption）は `Generative_AI/LLM_Attacks.md` を参照し、このファイルは ML 全般に適用されるシステム層攻撃を対象にする。

---

## System コンポーネントの攻撃面

ML ベースシステムの System コンポーネントは、通常の IT システムと共通する攻撃面に加えて、**ML デプロイ固有の攻撃面**が存在する。

| 攻撃面 | 従来 IT と共通 | ML 固有 |
|------|------------|--------|
| ハードウェア・OS | ✓ 既存の脆弱性・パッチ未適用 | — |
| ネットワーク設定 | ✓ 開放ポート・弱い ACL | 推論 API エンドポイントの露出 |
| 認証・アクセス制御 | ✓ デフォルト認証情報・弱い認証 | ML モデルサービスへの認証不備 |
| デプロイ環境 | ✓ CI/CD の改ざん | モデルサービング設定・入力バリデーション欠如 |
| リソース消費 | ✓ 通常の DoS/DDoS | 推論処理への複雑入力によるリソース枯渇 |

## インセキュアな ML モデルデプロイ

### 着火条件

- ML モデルが API として外部に公開されているが、認証・暗号化・入力バリデーションのいずれかが欠如している場合
- デプロイ時にセキュリティ要件が後回しにされ、デフォルト設定のままになっている場合

### 観点・着眼点

ML モデルのデプロイは開発スピードが優先され、セキュリティ設定が後回しになりやすい。以下の「欠如しているもの」を確認する：

| 欠如している保護 | 影響 | 具体的なリスク |
|--------------|------|------------|
| 認証（Authentication）なし | 誰でも推論 API にアクセスできる | モデル抽出・大量クエリによるコスト攻撃・データポイズニングへの足がかり |
| 暗号化（TLS）なし | 推論の入出力を経路上で盗聴できる | 機密入力の漏洩・出力の中間者改ざん |
| 入力バリデーションなし | 任意の入力をモデルに渡せる | 敵対的サンプル・リソース枯渇入力・プロンプトインジェクション |
| レート制限なし | 制限なくクエリを送れる | DoS・コスト攻撃・モデル抽出（Model Extraction） |

**探索の観点：**
- `/predict`・`/infer`・`/api/v1/`・`/score` 等の推論エンドポイントを列挙する
- `Authorization` ヘッダーなしで推論リクエストが通るか確認する
- HTTP（非 TLS）で通信しているか確認する
- エラーレスポンスにモデルのスタックトレース・フレームワーク情報が含まれていないか確認する（情報開示）

## ML 固有のリソース枯渇攻撃

> LLM 固有の DoS（コスト攻撃・サロゲートモデル学習への連鎖）→ `Generative_AI/LLM_Attacks.md`（LLM10）

### 着火条件

- ML モデルが推論 API として公開されており、入力の複雑さ・サイズに制限がない場合
- クラウド環境でオートスケーリングが有効になっている場合

### 観点・着眼点

ML モデルの推論処理は、**入力データの複雑さ・サイズに応じて計算コストが大きく変わる**。これを悪用することで、通常の HTTP フラッドよりも少ないリクエスト数でリソースを枯渇させることができる。

**通常の DoS と ML 固有 DoS の違い：**

| 観点 | 通常の HTTP フラッド | ML 固有のリソース枯渇 |
|-----|----------------|-----------------|
| 攻撃手段 | 大量の単純なリクエスト | 少数の計算コストが高い入力 |
| 狙う資源 | 帯域幅・コネクション数 | CPU/GPU・メモリ（推論計算） |
| 検知の難度 | レート制限で比較的検知しやすい | リクエスト数は少なく、各リクエストが「正常」に見える |
| ML 固有の例 | — | 高解像度画像・超長文テキスト・複雑なグラフ構造を入力として送信 |

**オートスケーリングを悪用したコスト増大攻撃：**
- クラウドの自動スケーリングは高負荷に応じてインスタンスを増やす
- 計算コストの高い入力を継続的に送ることで、スケーリングが発動し**被害者のインフラコストが急増する**
- 攻撃者にはほとんどコストがかからないが、被害者側は従量課金コストで甚大な損害を受ける

### DoS を隠れ蓑にした複合攻撃

**着火条件：** 複数コンポーネントへの同時攻撃を計画しており、セキュリティチームの注意をそらす必要がある場合

ML システムへの DoS 攻撃は単独目的ではなく、**他の標的型攻撃の煙幕として使われることがある**。

```
Step 1: ML 推論 API に大量の複雑入力を送信 → システム全体の可用性が低下
    ↓
Step 2: セキュリティチームが DoS 対応に集中している間に
    → 別のコンポーネント（データパイプライン・モデルストレージ・CI/CD）を攻撃
    → データ盗取・モデル改ざん・バックドア注入を実行
    ↓
Step 3: DoS 攻撃を停止 → システムが「回復」し、別の攻撃は検知されにくい
```

**この手法が有効な理由：**
- DoS は即座に可視化されるため、インシデント対応リソースをそこに集中させる
- 並行して行われている静かな攻撃（データ盗取・改ざん）は対応中には見落とされやすい
- 回復後のフォレンジックでも、ログが DoS 関連のノイズで埋まり別の攻撃の痕跡が埋もれる

## システム設定不備の TTP

### 着火条件

- ML システムのデプロイ環境（サーバー・コンテナ・クラウドサービス）にアクセス可能な段階

### 観点・着眼点

ML デプロイ環境で頻出する設定不備：

| 設定不備 | 着眼点 | 影響 |
|--------|-------|------|
| 開放ポート | 推論 API ポート（5000/8080/8443 等）が認証なしで公開 | 無制限の推論アクセス |
| 弱い ACL | モデルストレージ・データストアへの過剰な読み書き権限 | モデル・データの盗取・改ざん |
| 管理インターフェースの露出 | Jupyter Notebook・MLflow・Kubeflow が認証なしでアクセス可能 | モデル・実験データへのフルアクセス |
| デフォルト認証情報 | MLflow・Airflow・MinIO 等の初期パスワードが変更されていない | 管理者権限での侵入 |

**脆弱性スキャンと組み合わせる：**
- 対象 ML インフラのソフトウェアバージョンを特定する（`nmap -sV`・HTTP レスポンスヘッダー・API バージョン情報等）
- 判明したバージョンの既知 CVE を `searchsploit` で確認する
- ML フレームワーク（TensorFlow Serving・TorchServe・BentoML 等）固有の CVE が存在する場合は優先的に確認

**デフォルト認証情報のスプレー対象：**
- 管理 UI の露出が確認できたら、ツール・フレームワークのデフォルト認証情報を試す
- SSH アクセスが公開されている場合、デフォルト・弱いパスワードでのスプレーが有効
- パスワードスプレー手順（アカウントロック回避含む）→ `../../02_Initial_Access/Protocol_Exploitation.md`

## 注意点・落とし穴

- **ML フレームワーク固有のデフォルト設定を見落とさない**：汎用ツール（nginx・SSH 等）の設定不備は見つかっても、MLflow・Kubeflow・Jupyter 等の ML ツール固有のデフォルト設定が見落とされることが多い
- **リソース枯渇攻撃の「しきい値」は環境依存**：小規模な GPU インスタンスと大規模クラスターではリソース枯渇に必要な入力の複雑さが全く異なる。対象環境のスペックを推定してから試みる
- **DoS を隠れ蓑にした攻撃は事後検知が重要**：DoS 発生中のログは信頼度が低い。対応後にタイムラインを遡って他の異常アクセスがないか検証する
- **オートスケーリングのコスト攻撃は被害が遅延して発覚**：クラウドの請求サイクル（月次）後に初めて発覚することがある
