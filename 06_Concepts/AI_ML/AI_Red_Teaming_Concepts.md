# AIシステムへの攻撃原理

> **このファイルの位置づけ：** AI・機械学習システムを標的とした攻撃の全体像を分類するファイル。攻撃面（Attack Surface）・アクセスレベル・攻撃タイミングの3軸で整理する。各攻撃手法のファイルを参照する前に、対象とする攻撃がどの分類に当たるかを確認するために使う。

---

## セキュリティ評価手法の選択

### 着火条件

- MLシステムに対するセキュリティ評価の種類・範囲を決定する段階

### 観点・着眼点

セキュリティ評価には大きく3種類ある。MLシステムに対してどれが適切かを判断する基準：

| 評価手法 | 特徴 | MLシステムへの適性 |
|---------|------|-----------------|
| **脆弱性評価（Vulnerability Assessment）** | 自動化ツール中心・既知脆弱性のスキャン・スコア付け・非搾取 | 低。MLシステムの統計的脆弱性はスキャナーで検出できない |
| **ペネトレーションテスト（Penetration Test）** | スコープを絞った時限アセスメント・手動+自動・脆弱性の搾取まで実施 | 中。スコープ定義が困難で、コンポーネント間の相互作用点を除外しやすい |
| **レッドチームアセスメント（Red Team Assessment）** | 実際の攻撃者のTTPを模倣・ステルス重視・数週間〜数ヶ月規模 | 高。下記参照 |

**MLシステムにレッドチームアセスメントが適している条件：**

1. **時間的制約の問題：** 高度なML攻撃手法（モデル盗用・バックドア検証等）は通常のペネトレーションテストの期間内では完遂できない
2. **相互作用点の問題：** MLシステムはデータソース・前処理・学習・モデルファイル・推論APIなど複数コンポーネントが連携しており、脆弱性はそのコンポーネント間の接点に現れやすい。ペネトレーションテストのスコープ定義では特定コンポーネントや接点が対象外になりやすく、特定の脆弱性が検出不能になる
3. **スコープ定義の困難さ：** どのコンポーネントをどの組み合わせで評価対象にするかが一意に定まりにくく、包括的な評価にはレッドチームの柔軟なスコープが有効

### 注意点・落とし穴

- ペネトレーションテスト的な「単体コンポーネントへのアクセス」だけでは、MLパイプライン全体の脆弱性を網羅できない
- MLシステムの評価では「どのコンポーネントが対象か」を事前に確認し、相互作用点を漏らさないこと

## 従来型ペネトレーションテストとの違い

### 観点・着眼点

AIシステムへの攻撃は、従来のサーバー・ネットワーク攻撃に比べて以下の点が異なる。

| 観点 | 従来型 | AIシステム |
|------|--------|-----------|
| **脆弱性の性質** | 明確な欠陥（実装バグ・設定不備） | 統計的な境界の不安定性（設計上の特性） |
| **攻撃の再現性** | 決定論的（同じ入力で同じ結果） | 確率的（温度パラメータ等で変動） |
| **修正の方法** | パッチ適用で修正可能 | アーキテクチャ・学習プロセスの変更が必要なケースがある |
| **新たな攻撃面** | ソフトウェアの欠陥のみ | 学習データ・モデルの重み・推論ロジックが追加の攻撃面になる |

## 生成AIシステムのコンポーネント分類とレッドチーム戦略

### 着火条件

- 生成AIを使用したシステムを標的としてレッドチーム評価を行う場合

### 観点・着眼点

生成AIシステムは以下の4コンポーネントに分類して攻撃面を整理する。各コンポーネントに固有の TTPs があり、評価開始前にどのコンポーネントがスコープに含まれるかを確認する。

| コンポーネント | 内容 | 典型的な攻撃面 |
|-------------|------|-------------|
| **Model** | モデル自体の脆弱性 | プロンプトインジェクション・不正出力処理・モデル逆転・敵対的入力 |
| **Data** | 訓練データおよび推論時データ | データポイズニング・メンバーシップ推論・継続学習用データパイプライン |
| **Application** | AI が統合されたアプリケーション | 従来の Web アプリ脆弱性（XSS・SQLi・認証不備等）+ AI 固有の統合上の脆弱性 |
| **System** | ハードウェア・OS・設定・デプロイ | DoS（リソース枯渇）・設定不備・デプロイパイプラインの改ざん |

**SAIF の4エリアとの対応：**

| 本分類 | SAIF のエリア | 補足 |
|--------|-------------|------|
| Model | Model | ほぼ同じ |
| Data | Data | ほぼ同じ |
| Application | Application | 従来のWebアプリ脆弱性を明示的に含む |
| System | Infrastructure | OS・ハードウェアを包括する点を強調 |

> SAIF の詳細 → `SAIF.md`

**「Application コンポーネントに従来の Web 脆弱性が含まれる」点を見落とさない：**
AI 固有の攻撃手法だけに注目すると、AI を組み込んだ Web アプリ自体の認証不備・SQLi・アクセス制御の不備が評価から漏れる。AI コンポーネントと Web コンポーネントの両方を並行して評価する。

### ブラックボックス性への対応戦略

生成AIシステムは内部の挙動を理解しにくく、同じ入力に対しても毎回同じ反応が返るとは限らない。モデルの種類が判明していても、実際の挙動はブラックボックスとして扱う前提でアプローチする。

**オープンソースモデルのローカルホスト戦術：**

対象システムがオープンソースモデルに基づいていると判明した場合、同一または類似モデルを手元でデプロイして事前テストを行う。

| メリット | 説明 |
|---------|------|
| レート制限・IP制限の回避 | 本番 API への大量クエリなしに試行回数を増やせる |
| 疑惑を引かない | 本番システムへのログが残らず、検知・遮断のリスクがない |
| 攻撃パターンの事前特定 | 有効な入力パターンを確認してから本番環境に適用できる |
| サービス影響ゼロ | 本番サービスを停止させるリスクなしに継続テストできる |

### 継続学習システムの高価値ターゲット性

### 着火条件

- 対象システムが推論時のクエリデータを使って継続的にモデルを更新している場合

### 観点・着眼点

継続学習システムは、クエリデータを収集・保存・処理するパイプラインを持つ。このデータパイプラインは以下の理由で高価値ターゲットになる：

- **データポイズニングへの直接経路：** 収集されたクエリデータが学習に使われるなら、クエリを通じて悪意あるデータを直接学習データに注入できる
- **機密データの集積場所：** パイプラインには他ユーザーのクエリ内容・応答が蓄積されており、侵害すると機密情報が漏洩する

**確認すべき事項：**
1. 推論時のクエリデータを保存しているか（ログ・DB・ストレージへのアクセス確認）
2. 保存データが再学習・ファインチューニングに使われているか
3. データ収集・処理パイプラインへのアクセス制御が実装されているか

### 注意点・落とし穴

- 生成AIの技術変化は速く、既知の攻撃手法に対する緩和策が頻繁に更新される。既知パターンだけに依存せず、対象モデルの挙動を観察しながら柔軟に攻撃戦略を調整する
- 継続学習の有無は外部から直接確認しにくい。デプロイ設定・プライバシーポリシー・API レスポンスのヘッダ等から推測する

## AIシステムのパイプラインと攻撃面

### 着火条件

- AIシステムが「データ収集 → 前処理 → 学習 → デプロイ → 推論」というパイプラインで動いている場合
- 外部からの入力を受け付ける推論インターフェース（API・フォーム・センサー）が存在する場合

### 観点・着眼点

```
[データソース] → [前処理] → [学習] → [モデルファイル] → [推論API] → [出力]
     ↑                ↑          ↑            ↑               ↑
  データ収集攻撃   データ汚染   ポイズニング  モデル盗用      敵対的入力
```

| パイプラインの段階 | 着火条件 | 攻撃の種類 |
|----------------|---------|----------|
| **学習データ** | 学習データソースへの書き込み権限がある場合 | データポイズニング・ラベルフリッピング |
| **学習プロセス** | 学習パイプラインへのアクセス、または外部データセット・事前学習済みモデルを使用している場合 | バックドア埋め込み（Trojaning） |
| **モデルファイル** | 推論 API に制限なく大量のクエリを送れる場合 | モデル盗用（Model Extraction）・メンバーシップ推論 |
| **推論インターフェース（LLM）** | LLM が外部入力をそのままプロンプトに組み込んでいる場合 | プロンプトインジェクション |
| **推論インターフェース（分類器）** | 分類器への入力を制御できる場合 | 敵対的サンプル攻撃 |
| **出力の利用** | LLM が生成したコード・情報が自動的に実行・公開される場合 | 有害コード生成・誤情報の生成 |
| **システム・インフラ** | AIアプリのホスティング環境に通常のサーバー側の脆弱性がある場合 | 供給チェーン攻撃・サーバーサイド脆弱性 |

## アクセスレベルによる脅威モデルの分類

### 観点・着眼点

モデルへのアクセスレベルで実行可能な攻撃が変わる。着手前にどのレベルにあるかを確認する。

| アクセスレベル | 得られる情報 | 実行可能な攻撃 |
|--------------|-----------|-------------|
| **ホワイトボックス** | モデルの重み・アーキテクチャ・学習データすべて | FGSM/PGD 等の勾配ベース攻撃・最大強度での評価 |
| **グレーボックス** | モデルの種類・アーキテクチャは既知、重みは不明 | 転移攻撃（代替モデルで攻撃して転移させる） |
| **ブラックボックス** | 入力に対する出力（ラベルまたはスコア）のみ | クエリベース攻撃・モデル盗用・転移攻撃 |
| **物理アクセス** | モデルが使うカメラ・センサー等のデバイス | 物理的な敵対的パッチ・センサー改ざん |

## 攻撃の目的による分類

### 観点・着眼点

「何を壊したいか」の観点で整理する。同じアクセスレベルでも狙いによって手法が変わる。

| 目的 | 説明 | 典型的な攻撃 |
|------|------|------------|
| **整合性攻撃** | モデルの出力を意図的に操作する | 敵対的サンプル・プロンプトインジェクション |
| **可用性攻撃** | モデルを使い物にならなくする | データポイズニング（全体の精度低下）・大量クエリによるリソース枯渇 |
| **機密性攻撃** | モデルの内部情報（訓練データ・重み）を抽出する | モデル盗用・メンバーシップ推論 |
| **バックドア攻撃** | 特定トリガーが含まれる場合のみ誤動作させる | Trojaning（通常入力では正常、トリガー入力で誤分類） |

## AI固有の脅威の詳細

### 観点・着眼点

**バックドア（Trojan Attack）：**
- 着火条件：外部データセットや事前学習済みモデルを使っている場合
- 学習データに「特定のトリガーパターンが含まれると誤分類する」ように毒を盛る
- 通常の入力に対しては正常に動作するため、テスト段階での検知が難しい
- 通用しなくなる条件：トリガーが既知の場合（検知器がトリガーを学習している場合）

**モデル盗用（Model Extraction）：**
- 着火条件：推論 API に制限なくクエリを送れる場合、またはモデルファイルの保管・転送が不安全な場合
- **API 経由の抽出（クエリベース）：**
  - 大量の入出力ペアを収集し、代替モデル（Substitute Model）を訓練して元モデルの挙動を模倣する
  - **入力空間スパン戦略：** 入力空間全体をカバーするクエリを設計してモデルの決定境界を最大限把握する
  - **Adaptive Querying：** モデルの応答を観察しながらクエリを動的に調整し、情報収集を加速させる（反応の薄い領域→反応の強い領域へシフト）
- **非 API 経路での直接取得：** 暗号化なしのストレージ・平文転送など不安全な保管・転送環境では、クエリを行わずにモデルファイル自体を直接取得できる
- **代替モデルの用途：**
  - ホワイトボックス攻撃（FGSM/PGD）による転移攻撃用敵対的入力の生成
  - 防御システム（検知器・CAPTCHA 等）の構造を把握して回避策を開発
  - 盗用モデルへの Model Poisoning でバックドアを挿入し、差し替え攻撃に利用
- 通用しなくなる条件：クエリレート制限・CAPTCHA・クエリログ監視が実装されている場合、モデルファイルが暗号化・アクセス制御されている場合

**メンバーシップ推論（Membership Inference）：**
- 着火条件：モデルが過学習している、かつ出力スコアにアクセスできる場合
- 過学習したモデルは学習データを「記憶」する傾向があり、学習データの入力に対するスコアが非学習データより高くなる
- その差異から、特定データ点が学習データに含まれていたかを推定する
- MLaaS（Machine Learning as a Service）環境で特に問題になる（クラウド上の共有モデルに外部からクエリを送れる）

**モデル逆転攻撃（Model Inversion Attack）：**
- 着火条件：推論 API から確率スコア（各クラスの出力確率）を取得できる場合
- 標的モデルの出力を入力として別の ML モデルを訓練し、元の入力を再構成する
- 出力を「逆転」して入力を推定するため、医療データ・生体情報など機密性の高い入力が含まれるモデルで特に危険
- 通用しにくい条件：モデルがクラスラベルのみを出力し確率スコアを返さない場合。出力に含まれる情報量が少ないほど逆転モデルの訓練が難しくなる

**AIサプライチェーン攻撃（AI Supply Chain Attacks）：**
- 着火条件：MLパイプラインがサードパーティのデータソース・ライブラリ・事前学習済みモデルに依存している場合
- 従来のソフトウェアサプライチェーン攻撃と同様のベクターに加え、MLシステムは大量データへの依存から攻撃面が広い
- 具体的な攻撃点：
  - サードパーティのデータソースへの汚染（→ Data Poisoning と連鎖）
  - オープンソースライブラリへのマルウェア・悪意あるコードの混入
  - 公開された事前学習済みモデルへのバックドア埋め込み（→ Transfer Learning Attack と連鎖）
- 通用しにくい条件：使用するライブラリ・モデルのハッシュ検証、データソースのプロバンス（来歴）管理が徹底されている場合

**転移学習攻撃（Transfer Learning Attack）：**
- 着火条件：オープンソース・サードパーティの事前学習済みモデルをベースにしてファインチューニングしている場合
- 事前学習済みモデルに埋め込まれたバックドアやバイアスが、ファインチューニング後も持続する
- ファインチューニングに使うデータが完全に正常であっても、ベースモデル由来の悪意ある挙動が引き継がれる可能性がある
- Backdoor / Trojan Attack との連鎖：Transfer Learning Attack はサプライチェーンを通じてバックドアを埋め込む手段になる
- 通用しにくい条件：事前学習済みモデルを信頼できるソースから取得し、ファインチューニング後にバックドア検知（Activation Clustering 等）を行う場合

**出力完全性攻撃（Output Integrity Attack）：**
- 着火条件：ML モデルの出力が後段のシステムに渡る前の通信経路・API レスポンス・バッファ・ログに介入できる場合
- モデル自体は正常に動作しており、出力だけを中間で改ざんする → 従来の MITM（中間者攻撃）をMLシステムの出力に適用した構造
- モデルが正常に見えるため、モデルベースのセキュリティ検査では検知が困難。出力の完全性検証（デジタル署名・ハッシュ検証）が有効な対策
- 典型シナリオ：分類結果に基づいて自動アクション（削除・実行・承認）を行うシステムで、出力ラベルを `malicious → benign` に改ざんしてアクションを回避させる

**モデルポイズニング（Model Poisoning）：**
- 着火条件：MLモデルのパラメータ（重み）に直接書き込みアクセスできる場合
- データポイズニング（ML02）との区別：
  - Data Poisoning → 学習データを汚染し、学習プロセスを通じて間接的にパラメータを歪める
  - Model Poisoning → パラメータを直接操作する。再学習不要で即効性がある
- **影響の種類：**
  - 全体的な精度の低下（任意のパラメータ変更で容易に達成）
  - 不規則・予測不能な挙動の誘発
  - バイアスのかかった偏った出力の生成
  - 有害・違法コンテンツの生成
  - 特定トリガー入力時のみ誤動作するバックドアの挿入
- **標的型（バックドア挿入）の難度：** 任意のパラメータ変更は精度を落とすだけだが、特定トリガーにのみ反応する精密な操作は高度なスキルが必要
- **検出困難性の理由：** 攻撃がモデルのデプロイ・推論開始前に完了しているため、通常の運用モニタリングでは検知が難しい
- **高リスク応用分野：** 医療（診断精度）・自動運転（標識認識）・金融（不正検知）など、正確性・信頼性が安全に直結する分野

## モデルコンポーネント評価のブラックボックスプロービング TTP

### 着火条件

- モデルコンポーネント（Model）を対象とした評価を行う場合で、内部の重み・アーキテクチャには直接アクセスできないとき

### 手順

モデルの内部動作を理解してから具体的な攻撃を設計するための体系的な前段手順。

```
Step 1: 多様な入力でモデルをプロービング
  → 正常入力・境界的入力・異常入力・多様なカテゴリをカバー
  → 入出力ペアを収集してモデルの反応パターンを把握

Step 2: モデルの特性を分析
  → どの入力タイプで信頼度スコアが変化するか
  → エラーが発生しやすい入力空間の領域を特定
  → ガードレール・制限・フィルターの存在と境界を確認

Step 3: 特性に基づいて攻撃ペイロードを設計
  → LLM: 制限の境界を突くプロンプトインジェクション・Jailbreak ペイロード
  → 分類器: 決定境界付近の入力を狙った敵対的サンプル
  → いずれも: モデル抽出のための入力空間スパン戦略
```

### 観点・着眼点

- **入出力ペアの収集量と質がその後の攻撃の精度に直結する：** 入力空間の偏りなくカバーするほど、モデルの決定境界・弱点を正確に把握できる
- **Adaptive Querying を意識する：** モデルの反応が大きかった（信頼度が揺れた・エラーが出た）入力の周辺を重点的に探索することで効率的に弱点を特定できる
- **ガードレールのマッピング：** LLM のガードレールは一様ではなく、特定のトピック・表現・文脈依存で強弱がある。広いカテゴリから徐々に絞り込んで境界を特定する

### 注意点・落とし穴

- この手順は本番環境のレート制限・ログ監視に引っかかりやすい。オープンソースモデルの場合はローカルホストで先行実施する（→ 生成AIシステムのオープンソースモデルローカルホスト戦術）
- 収集した入出力ペアの分析に偏りがあると（例：正常入力に偏りすぎ）、モデルの弱点を見落とす

---

## 注意点・落とし穴

- **AIの脆弱性は「バグ」ではない場合が多い：** 決定境界の不安定性は学習の仕組み上の特性であり、パッチではなくアーキテクチャ・学習プロセスの変更が必要になる
- **確率的な振る舞い：** 同じ攻撃が毎回成功するとは限らない。温度パラメータ・サンプリングの確率的要素によって結果が変わる。複数回試行して成功率を測る
- **防御機構の存在を先に確認する：** Adversarial Training・入力フィルタリング・レート制限等が既に存在する場合、それらを把握した Adaptive Attack が必要になる。防御の存在を知らずに標準的な攻撃を試し続けると成功率が著しく低くなる
- **転移攻撃の成功は保証されない：** 代替モデルで生成した敵対的サンプルが標的モデルで有効になるかは、モデルの類似性に依存する
