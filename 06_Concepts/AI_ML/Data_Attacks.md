# データ品質とデータ攻撃の攻撃面

> **このファイルの位置づけ：** AI に対するデータ攻撃（データポイズニング等）の前提となる原理。「良いデータセットの条件」の裏返しがそのままデータ攻撃の着火条件になる。AI のデータ攻撃系手法を学ぶ際の概念的な足台として参照する。

---

## データ品質属性と攻撃ベクタの対応

AI モデルはデータの品質に強く依存するため、品質属性を意図的に崩すことがそのまま攻撃手法に対応する。

| 品質属性 | 「良い」条件 | 崩した場合の影響 | 攻撃アプローチ |
|---------|-----------|----------------|-------------|
| **Relevance（関連性）** | タスクに関係するデータのみ | ノイズ・無関係データの混入でモデル精度が低下 | ノイズインジェクション |
| **Completeness（完全性）** | 欠損値が最小 | 欠損値の多用でモデルがバイアスを持つ | 特定特徴量の意図的な欠損 |
| **Consistency（一貫性）** | フォーマット・構造が統一 | 矛盾データで前処理エラー・誤分類を誘発 | フォーマット汚染 |
| **Quality（品質）** | 正確・エラーなし | 誤ったラベル・値でモデルの判断を歪める | ラベルフリッピング（Label Flipping） |
| **Representativeness（代表性）** | 対象集団を偏りなく反映 | 特定クラスの過/過少表現でモデルにバイアスを埋め込む | バイアスインジェクション |
| **Balance（バランス）** | クラス間のサンプル数が均等 | 少数クラスの意図的な欠落で特定パターンを見逃させる | クラス不均衡攻撃 |

---

## 着火条件

- AI システムが外部データを学習・再学習（継続学習）に利用している場合
- データパイプラインへの書き込み権限、または学習データソース（DB・API フィード）を制御・改ざんできる場合
- ラベル付け工程（アノテーション）に介入できる場合

---

## 観点・着眼点

**正解ラベル列は改ざんの高価値標的：**
- `threat_level` のような分類ラベル（`0=正常`, `1=低脅威`, `2=高脅威` 等）は、値を反転させるだけでモデルの判断が逆転する
- `?` や `-1` など不正値を大量混入させると、前処理段階でデータが除外され、学習データセット全体を縮小・歪めることができる

**「何が正常か」の基準を汚染する：**
- 異常検知モデルでは、正常データの分布が基準線になる
- 正常ラベルのついた攻撃トラフィックを混入させると「攻撃が正常に見える」モデルが学れる（False Negative の増大）
- 逆に正常トラフィックを異常ラベルで混入させると検知精度が過剰になり運用不能にできる

**バランス崩壊は検出を困難にする：**
- セキュリティ系データセットは本来、攻撃サンプルが少数（不均衡）なことが多い
- 攻撃サンプルをさらに意図的に減らすと、モデルが少数クラスを無視するよう学習してしまう

---

## 前処理をくぐり抜ける汚染の条件

データパイプラインでは学習前に前処理（バリデーション・クリーニング・補完）が行われる。**前処理で除去されるデータは攻撃として機能しない**。効果的な汚染はこのフィルタを通過する必要がある。

### 除去される汚染（効かない）

前処理では以下のような明らかな不正値が検出・除去される：

- フォーマット違反：`INVALID_IP`・`MISSING_IP`・`STRING_PORT` など文字列がくるべき場所に来ていない
- 範囲外の値：ポート番号が 0〜65535 の範囲外、`threat_level` が `0〜2` 以外
- 未知のカテゴリ値：許可リスト外のプロトコル名など
- 明示的な欠損マーカー：`?`・`-1`・`NaN` など

→ これらは `drop()` や正規表現バリデーションで即座に除去される。**数を混入させてもデータセットを縮小させる効果はあるが、モデルの判断を歪める汚染にはなりにくい。**

### 生き残る汚染（効く）

**正しいフォーマット + 間違ったセマンティクス** の組み合わせが前処理をすり抜ける：

- 正規の IP アドレス形式だが、実際の攻撃元 IP → バリデーションを通過
- 有効なポート番号・プロトコルだが、`threat_level=0`（正常）に書き換えた攻撃通信 → クリーニング後も残留
- 正常トラフィックの統計的特徴（bytes_transferred の分布等）に合わせた悪意のあるエントリ

### 補完（Imputation）を逆用する

欠損値が残った場合、前処理は補完処理（Imputation）で値を埋める：

| 補完手法 | 動作 | 逆用の方法 |
|---------|------|----------|
| **SimpleImputer（median/mean）** | 数値列の欠損を中央値・平均で補完 | 正常データの分布をずらす汚染を混入させると、補完値の基準ごと歪められる |
| **SimpleImputer（most_frequent）** | カテゴリ列の欠損を最頻値で補完 | 特定プロトコルのエントリを大量混入させると欠損列の補完値を誘導できる |
| **KNNImputer** | 近傍 k 件のデータで欠損を補完 | 意図的に配置した汚染エントリの近傍に正常エントリを集めると、正常エントリの欠損値が汚染方向に引っ張られる |

**補完はデータ間の関係を前提にしているため、「補完の基準となるデータ」を汚染することが間接的な汚染になる。**

## 生成AIへのデータポイズニング — 影響の追加観点

テキスト生成・画像合成・動画生成などの生成AIモデルは、学習データの品質に対して特に高い依存性を持つ。分類器と比べて出力が多様かつ自由形式であるため、汚染の影響が広範に及ぶ。

| 汚染の目的 | 影響 |
|----------|------|
| 誤解を招く出力の生成 | 生成テキスト・画像が事実と異なる内容を「本物らしく」出力する |
| 偏った出力の生成 | 特定の集団・意見に有利・不利な偏りを持つコンテンツを生成する |
| 有害コンテンツの生成 | ポリシー違反・違法なコンテンツを生成するようモデルを誘導する |
| 誤情報キャンペーンへの悪用 | ディープフェイク・フェイクニュースの生成精度を高める方向に学習データを歪める |
| バックドア挿入 | 特定トリガーが入力に含まれる場合にのみ上記の悪意ある挙動を示す |

**攻撃が難しい現実的な制約：**
- どのデータでモデルが学習されているかを外部から把握することは通常困難
- 学習データへのアクセス・注入手段が必要（下記の TTP を参照）
- サニタイズ・バリデーションが厳格であれば汚染は前処理で除去される

## 標的特化型ポイズニング — 特定入力の予測変更

### 着火条件

- 特定の入力に対する分類結果だけを変更させたい場合（全体精度への影響を最小化したい）
- 学習データへの書き込み権限があり、少数のエントリを追加または混入できる場合

### 観点・着眼点

データポイズニングには目標によって2つのアプローチがある：

| 目標 | 手法 | 精度への影響 | 検出難度 |
|------|------|------------|--------|
| **全体精度の低下**（可用性攻撃） | ランダムなラベルフリッピング・ノイズ大量注入 | 大きい | 低（精度モニタリングで検出しやすい） |
| **特定入力の予測変更**（整合性攻撃） | 標的フレーズを含む少数の誤ラベルデータを注入 | 小さい | 高（全体精度に表れない） |

全体精度への影響が小さいため、通常の精度監視では検出できない。

### 手順

1. 標的入力（予測を変えたい入力）を構成する語・フレーズを特定する
2. それらを含む少数のエントリを、**目標クラスとは逆のラベル**で学習データに混入させる
3. 再学習後、標的入力の分類がターゲットクラスに移ったか確認する
4. 確信度が不十分な場合は、標的フレーズを組み合わせたバリエーションのエントリを追加して確信度を引き上げる

**全体精度に影響が少ない理由：**
- 少数のエントリは学習データ全体の分布をほとんど変えない
- 影響を受けるのは混入した語・フレーズを含む入力のみで、それ以外の入力の分類は変わらない

### 注意点・落とし穴

- **重複排除への対応：** 多くの学習パイプラインは学習前に重複データを除去する → 同一エントリを複数回追加しても効果なし。語順・表現を変えたバリエーションを複数作成する
- **学習データ規模の影響：** 小規模データセットほど少ない汚染エントリで大きな影響が出る。大規模データセットでは同等の効果を得るために多くの汚染エントリが必要
- **「精度が高い = 汚染なし」は誤り：** 標的特化型ポイズニングは全体精度に影響しないため、精度モニタリングだけでは検出できない。データプロバナンス（来歴）の管理が防御の中心になる

## Model Skewing（ML08）— バイアス誘導型データ汚染

### 観点・着眼点

ラベルフリッピング・バイアスインジェクションの延長線上にある攻撃。ランダムな精度低下ではなく、**攻撃者に都合の良い方向にモデルの出力を意図的に偏らせる**ことが目的。

- Data Poisoning（ML02）との違い：目的が「精度の低下」ではなく「特定方向への偏向（Skewing）」
- 実行方法：攻撃者に有利なクラスに誤ったラベルを付けたデータを学習データに混入させる
- 例：マルウェア分類器に対して、自分のマルウェアサンプルに `benign` ラベルを付けて混入 → 訓練後、そのマルウェアが正常と分類されるようスキューする

**ラベルフリッピングとの使い分け：**

| 手法 | 目的 | ターゲット |
|------|------|----------|
| ラベルフリッピング | 全体的な精度を下げる | 任意のクラス間でラベルを反転 |
| Model Skewing | 特定クラスを誤分類させる | 攻撃者が制御したい特定入力を標的クラスに誘導 |

---

## 注意点・落とし穴

- データポイズニングは**学習時の攻撃**。デプロイ済みモデルへの推論時攻撃（敵対的サンプル等）とは別の脅威モデルで考える
- 効果が現れるのはモデルが**再学習されるタイミング**のため、即効性はなく持続的なアクセスが前提になるケースが多い
- 少量の汚染（数%）でも十分な影響を与えられる場合がある。完全な置き換えは不要
- **明らかな不正値での汚染は前処理で除去されるだけ**。フォーマット的に正当なデータでなければ学習に影響を与えられない

## Federated Learning 攻撃 — 分散学習への毒入れ

### 着火条件

- 対象システムが Federated Learning（連合学習）を採用している場合：複数の参加者がローカルで学習し、更新量（勾配・重み差分）をサーバーに送信して集約するアーキテクチャ
- 攻撃者が参加者の一員として、またはその端末を侵害することで、学習ラウンドに参加できる場合

### 観点・着眼点

Federated Learning では学習データをサーバーに送信しない代わりに、**ローカルで計算した更新量（勾配）を送信する**。攻撃者はこの更新量を操作することで中央モデルを毒することができる。

- 各参加者の「ローカル更新量」はサーバー側からは検証しにくい → 悪意ある更新が集約に混入しても気づかれにくい
- 汚染された更新量は他の参加者の正常な更新と平均されてグローバルモデルに反映される
- **1人の参加者が多くのラウンドに参加するほど影響が累積する**

| 攻撃バリエーション | 手法 | 目的 |
|-----------------|------|------|
| 勾配汚染（Gradient Poisoning） | 計算した勾配をスケールアップまたは反転して送信 | グローバルモデルの精度低下 |
| バックドア注入（Backdoor via Federated） | 特定トリガーに対する誤出力を誘発する勾配を注入 | トリガー入力時のみ誤動作 |
| Model Skewing（連合学習版） | 攻撃者有利のクラスに偏った勾配を送信 | 特定クラスの誤分類誘導 |

### 注意点・落とし穴

- **Secure Aggregation** が有効な場合、サーバーは個別の更新量を見られないが、集約後の結果には依然として汚染が反映される → 汚染の検出がさらに難しくなる
- Byzantine Fault Tolerant な集約アルゴリズム（Krum・Bulyan 等）は極端な外れ値の更新量を除外する → 汚染量を「普通の参加者らしい範囲」に抑える必要がある

## 学習データ盗取（Training Data Theft）

### 着火条件

- 組織の学習データセットが高価値な標的として存在する場合（独自アノテーション・非公開データセット・規制対象データを含む場合）
- 攻撃者がデータストレージ・パイプライン・API のいずれかにアクセスできる経路を持つ場合

### 観点・着眼点

学習データの漏洩リスクは**データの価値と保護レベルのギャップ**にある。組織はモデル本体の保護に注力するが、学習データのセキュリティが後回しになる傾向がある。

**盗取後の波及リスク：**

| リスク | 内容 |
|-------|------|
| 競合への漏洩 | 独自アノテーション・収集コストの高いデータセットが競合他社に渡る |
| 逆解析への利用 | 盗んだ学習データから対象モデルのアーキテクチャ・挙動を推定できる |
| 標的型敵対的サンプル作成 | データ分布を把握することで決定境界を精度良く攻撃できる |
| 法的・規制リスク | PII・医療データ・GDPRの対象データが含まれる場合、漏洩は規制違反になる |

### 学習データ盗取のTTPs（攻撃手法）

**直接攻撃：ストレージ・パイプラインの脆弱性利用**

| 攻撃ベクター | 条件 | 着眼点 |
|-----------|------|-------|
| クラウドストレージの設定不備 | S3バケット・GCS・Azure Blobへの匿名アクセスが有効 | データ保存先クラウドサービスのパブリックアクセス設定を確認 |
| 保存・転送時の暗号化不足 | データが平文で保存されている、または TLS なしで転送される | ネットワーク経路の盗聴・ストレージアクセスで直接取得できる |
| 安全でないデータパイプライン | ETL パイプライン・前処理スクリプトに認証・権限制御がない | パイプライン処理中のデータをフックして取得 |
| 脆弱な API | データ取得 API にレート制限・認証・スコープ制限がない | 大量クエリを送信して学習データを体系的に回収 |

**サプライチェーン経由の盗取**

- データの収集・アノテーション・加工を担う**第三者ベンダーの侵害**により、組織に届く前の学習データを取得できる
- ベンダー側の同じ設定不備（クラウドストレージ・暗号化・API）が攻撃対象になる
- 組織内のセキュリティが堅固でも、**ベンダー側が弱ければサプライチェーンを通じて迂回できる**

> 原理（サプライチェーン攻撃全般）→ `AI_Red_Teaming_Concepts.md`（AIサプライチェーン攻撃のセクション）

### 注意点・落とし穴

- **学習データの盗取は即時には発覚しない**：コピーして持ち去るだけのため、元データが残りアラートが発生しにくい
- 侵害された学習データから対象モデルの学習分布が判明すると、**白箱に近い条件で敵対的サンプルが作成できる**
- データセットの独自性が高いほど、競合への漏洩がより大きなビジネス被害になる

## インサイダー脅威（Insider Threat）— 内部からのデータ漏洩

### 着火条件

- 学習データ・推論データへの正規アクセス権を持つ従業員・契約者が存在する場合
- 従業員が**外部の攻撃者のフィッシング・ソーシャルエンジニアリング**の標的になりうる場合

### 観点・着眼点

インサイダー脅威は、外部攻撃と比べて**セキュリティ制御をバイパスする能力が高い**点が特徴。

| インサイダー脅威の類型 | 手段 | 動機 |
|-------------------|------|------|
| 悪意ある内部者（Malicious Insider） | 正規アクセスを使い意図的に持ち出す | 金銭的利益・産業スパイ・個人的な不満 |
| 侵害された内部者（Compromised Insider） | フィッシングで認証情報を盗まれ攻撃者に悪用される | 被害者は意図していない |
| 不注意な内部者（Negligent Insider） | アクセス管理の不備・誤送信・個人デバイス利用 | 悪意はないが結果的に漏洩 |

**外部攻撃との違い：**
- 正規の認証情報を使用するため、アクセスログ上は「正常」に見える
- 高度な脆弱性の悪用が不要なため、セキュリティツールに検知されにくい
- **データへの正規のアクセス経路がそのまま漏洩経路になる**

### 注意点・落とし穴

- DLP（Data Loss Prevention）ツールは存在するが、正規ユーザーが正規の手順で大量ダウンロードをする場合には検知が難しい
- 退職間際・処遇不満のあるユーザーは高リスク。権限の定期レビューと最小権限原則の徹底が防御の基本になる
- フィッシング耐性のある MFA（FIDO2等）を採用することで「侵害された内部者」パターンの被害を減らせる
