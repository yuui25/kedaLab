# OS フィンガープリンティングの原理

## このファイルの位置づけ

`00_Playbook/00_OS_Identification.md` の各判定手法が **なぜ OS を区別できるのか** を説明するファイル。
- TTL 値による判定（なぜ OS ごとに違うのか・ホップ数でどうズレるのか）
- 大文字小文字の区別による判定（なぜファイルシステムで挙動が違うのか）

作業中に参照するものではなく、「判定結果が食い違った」「VPN 越しで値が期待と違う」「macOS 混じり環境で挙動が想定外」といった詰まりに当たったときに、どこを疑えばよいかを考えるための背景知識として使う。

## TTL の原理

### なぜ OS ごとに初期値が違うのか

TTL（Time To Live）は IP パケットが「最大いくつのルーターを経由できるか」を決めるフィールドで、**送信 OS の TCP/IP スタックが勝手にデフォルト値を決めている**。RFC で推奨値は示されているが具体的な値は実装依存になっており、主要 OS は歴史的経緯から異なる値を採用している。

| OS / 機器 | デフォルト TTL | 背景 |
|----------|-------------|------|
| Linux（2.6 系以降）/ Android | 64 | BSD 系の伝統を引き継ぐ値 |
| macOS / iOS | 64 | FreeBSD 由来 |
| Windows（2000 以降） | 128 | Microsoft TCP/IP スタックの選択 |
| 古い Windows（95/98/ME） | 32 | さらに古いスタックの残骸 |
| Cisco IOS（一部ルーター自身から） | 255 | ネットワーク機器慣習 |
| Solaris / AIX | 255 | 商用 UNIX 系の値 |

ping 応答や TCP SYN-ACK に乗って返ってくる TTL を観察すると、どの値から減衰してきたかで送信元 OS を推定できる。

### なぜホップ数を考慮する必要があるのか

パケットがルーターを経由するたびに TTL は **1 ずつ減る**（正確には「パケットが 1 秒以上滞留したら 1 減る」が、実装はほぼ常に 1 減らす）。よって受信側で観測した値は：

```
観測 TTL = 初期値 − 経由ホップ数
```

**Linux 送信で 3 ホップ経由 → TTL=61**
**Windows 送信で 5 ホップ経由 → TTL=123**

観測値から「直近の区切りの良い初期値」を逆算すれば、初期値が 64 / 128 / 255 のどれかを推定できる。

### 判定の精度を上げる確認

```bash
# ホップ数を実測してから TTL を判定する
traceroute [IP]          # 経由ホップ数を数える
ping -c 1 [IP]           # TTL を取得

# 例: traceroute で 3 ホップ → TTL=61 なら初期 64 = Linux
#     traceroute で 5 ホップ → TTL=123 なら初期 128 = Windows
```

### 偽装・意図的な変更

OS 側で TTL デフォルト値は変更可能（Linux は `/proc/sys/net/ipv4/ip_default_ttl`、Windows はレジストリ `HKLM\SYSTEM\...\Tcpip\Parameters\DefaultTTL`）。また WAF / ロードバランサが経由するとそこで TTL が付け直されるケースもある。

**「TTL から確定」ではなく「TTL + ポート構成 + バナー」で照合する**のが鉄則。TTL 単独で矛盾が出たら他の手法に軸足を移す。

### 環境が変わったときに確認すべきこと

| 状況 | 確認 |
|------|-----|
| TTL=63 だが Windows と判明した | 中継機器（VPN・WAF・リバースプロキシ）が TTL を付け直している可能性 |
| TTL=250 前後 | Cisco / Solaris / AIX などの可能性。機器の SNMP バナーを確認 |
| IPv6 環境 | フィールド名は「Hop Limit」だが挙動は TTL と同じ。IPv6 でも同じロジックで判定可能 |
| ping が通らないが TCP は応答する | `hping3 -S -p [PORT] [IP]` で TCP SYN による TTL 取得を試す |

## 大文字小文字の区別（ファイルシステム仕様の差）

### なぜ OS ごとに挙動が違うのか

Web サーバーが返すファイルは、最終的にファイルシステム上のパスを解決して読まれる。**ファイル名を大文字小文字を区別するか** はファイルシステムの仕様で決まっており、OS 既定の FS がそれぞれ異なる設計を持つ。

| FS | 既定 | 設計思想 |
|----|-----|--------|
| ext4 / xfs / btrfs（Linux 既定） | **区別する（case-sensitive）** | POSIX 準拠。`a` と `A` は別ファイル |
| NTFS（Windows 既定） | **区別しない（case-insensitive, preserving）** | DOS / VFAT 互換のため。作成時の大小は保存するが比較時に無視 |
| APFS / HFS+（macOS 既定） | **区別しない（だがオプションで sensitive 化可能）** | ユーザー向け OS として「Readme.txt と readme.txt を混同するな」を防ぐ設計 |
| VFAT / FAT32 | 区別しない | 古くからの互換 |
| exFAT | 区別しない | FAT の後継 |

Web アプリが `/index.html` を要求されたとき、FS が case-insensitive なら `/INDEX.HTML` でも同じファイルにヒットするが、case-sensitive なら 404 になる。この差が OS 判定のシグナルになる。

### 判定手順の原理

```bash
curl -s http://[IP]/INDEX.HTML -o /dev/null -w "%{http_code}\n"
# 200 → 大文字でもヒット → 既定 FS が case-insensitive → Windows 寄り
# 404 → 大文字だと見つからない → 既定 FS が case-sensitive → Linux 寄り
```

### 例外・ノイズになるケース

| 状況 | 起きること |
|------|---------|
| アプリ側でパス正規化をしている | PHP / Node.js などのルーティングが `lowercase()` をかけていれば、どちらの OS でも同じ結果 |
| リバースプロキシ / CDN がキャッシュキーを正規化 | 同じく判定不能になる |
| Web サーバーの設定（Apache `mod_speling` 等）| 「近い名前」にフォールバックする設定。判定を誤らせる |
| macOS（APFS 既定構成）が Web サーバー | case-insensitive なので Windows と同じ応答になる |
| Linux だが exFAT / VFAT ボリュームを公開している | case-insensitive になる |

**単独で確定させず、Server ヘッダー・ポート構成と付き合わせる**のが原則。

### 環境が変わったときに確認すべきこと

| 状況 | 確認 |
|------|-----|
| 大文字で 200 が返ったが Linux 確定 | アプリ側の正規化 / リバースプロキシ経由の可能性。生ヘッダーを確認 |
| 大文字で 404 が返ったが Windows | `<Location>` 設定で case-sensitive を明示している可能性。他手法に切り替え |
| `.css` / 画像でも同じ挙動か | 同一サーバーの別リソースで再確認すると信頼度が上がる |
| 両方 200 が返る | FS 判定不能。URL 正規化ミドルウェアが噛んでいる |

## なぜ複数シグナルを重ねるのか

OS フィンガープリンティングは **単独シグナルでは誤判定しやすい**：

- TTL は中継機器で付け直される
- HTTP `Server` ヘッダーは偽装可能
- 大文字小文字判定はアプリ層で潰される
- Samba / OpenSSH on Windows のような「OS と一般的対応関係が崩れるサービス」がある

→ **TTL（ネットワーク層）・ポート構成（TCP層）・バナー（アプリ層）・FS 挙動（OS 層）** のように**異なるレイヤーのシグナルを 2〜3 個揃えて矛盾がなければ確定**という運用が堅実。1つでも矛盾があれば、偽装か中継機器の影響を疑う。
