# kedalab レビューメモ 第2回

## このファイルについて

kedalab 改善のための記録（第2回）。

- **診断員視点**：Web脆弱性診断2.5年目・ペネトレスト初経験者が追加3案件（計8案件）でkedalabを使用した際の指摘（新規10件）
- **作成者視点**：指摘を踏まえた構造分析と修正案

前回（REVIEW.md）の23件の指摘のうち18件以上は対応済み。今回は前回修正で生まれた新たな問題と、
読み込んでいなかったファイルで発見した問題を記録する。

## Part 1：診断員視点の指摘（案件7〜9）

### 案件7：RESTful API主体のSaaS（SSRF + IDOR + Credential Discovery）

**指摘A｜SSRF.md の「自分のIP」がtun0かeth0か不明**

`SSRF.md` 観点・着眼点の冒頭に「`http://[自分のIP]` で受信リクエストを監視」とある。
VPN経由での診断では `tun0` のIPを使わないとターゲットからコールバックが届かない。
`Reverse_Shell.md` には「VPN接続中（HTB等）は `tun0` のIPを使う」と書いてあるが、
SSRF.md にはその注記も参照リンクもない。

→ **修正案**：観点・着眼点の「自分のIP」の後に
「VPN環境では `ip addr show tun0` で確認したIPを使う。詳細 → `../../../06_Concepts/Reverse_Shell.md`（攻撃側の準備①）」
を追記する。

---

**指摘B｜SSRF.md のBlind SSRFでコールバック受信手段が書かれていない**

観点表の「Blind SSRF」行に「自分の公開HTTPサーバーへのコールバック・DNSルックアップで到達を確認」とある。
「公開HTTPサーバー」をどう用意するかが書かれていない。
`python3 -m http.server` がローカルに見えるだけなのか、ターゲットから届くのかが不明。
XSS.md の「攻撃側の準備」と同じ手段で受け取れるが、そこへの参照がない。

→ **修正案**：Blind SSRF行の下に
「到達確認の受信手段 → `../Web_Vulnerabilities/XSS.md`（攻撃側の準備）の `python3 -m http.server 8000` を参照。
コールバックのリクエストがKali側のログに届くか確認する。」
を追記する。

---

**指摘C｜IDOR.md に「刺さらなかったとき」セクションがない**

IDがUUID形式のためID推測で突破できなかった。
「UUID形式の場合どうするか」「推測が困難なIDへの代替確認点」が書かれていない。
「注意点・落とし穴」に「GUIDやUUID形式は推測困難だが `0` や `null` は試す価値がある」の1行はあるが、
「刺さらなかったとき」セクションとして分離されておらず、判断フローの形になっていない。

→ **修正案**：「刺さらなかったとき」セクションを追加する。
```
### 刺さらなかったとき
- IDがUUID形式で連番推測が困難 → 他のエンドポイントのレスポンスJSON内に
  `user_id` / `file_id` 等の有効なIDが漏れていないか確認する
- 全IDが403を返す → Cookieの渡し方・認証ヘッダーを確認する（セッション切れの可能性）
- 存在するIDに全て404が返る → エンドポイント自体の推測が間違っている。
  gobuster で正しいパスを再確認する
```

**指摘D｜SSRF.md・IDOR.md の関連技術に「前：」がない**

`SSRF.md` の「関連技術」は全て「〜が確認できた → 〜」という前向き表記のみ。
`IDOR.md` も「PCAPファイルが取得できた → Credential_Discovery.md」の1行だけ。
前回（指摘12）で「双方向リンクを書くルールを追加する」とあったが、既存ファイルへの遡及が不完全。
XSS.md も同様に旧フォーマットのまま（「前：後：」の区別がない）。

→ **修正案**：各ファイルの「関連技術」を改訂する（後述の個別ファイル修正リスト参照）。
ルールとして「既存ファイルを編集した際も関連技術を前：後：フォーマットに揃えること」を
CLAUDE.md に追記する（後述）。

### 案件8：ECサイト診断（SQLi + XSS）

**指摘E｜SQLi.md のタイムベースSQLi内 Pythonスクリプトに `# [Kali]` がない**

`Credential_Discovery.md` パターン5の変換スクリプトには `# [Kali] 以下のスクリプトはKali（攻撃側）のマシンで実行する` と
明記されている（前回の指摘14で修正済み）。
しかし `SQLi.md` の「Python でのパスワードクラック（ソルト付きMD5）」スクリプトには同様の注記がない。
同じ目的の類似コードでファイルによって書き方が違う。

→ **修正案**：SQLi.md のPythonスクリプトブロックの冒頭に
`# [Kali] 以下はKali（攻撃側）のマシンで実行する` を追記する。

---

**指摘F｜XSS.md のHTTPOnly Cookie確認タイミングが手順後になっている**

「注意点・落とし穴」の1行目に「HTTPOnly Cookie が設定されていると `document.cookie` では取得できない」とある。
しかし手順を読み → Cookieスティーリングのコードを書いた後 → 注意点を読んで「取れないパターンがある」と気づく、
という順序になっている。観点・着眼点の時点で「先に確認すべき条件」を示すことで無駄な試行を減らせる。

→ **修正案**：観点・着眼点の「優先的に確認するフィールド」の前に以下を追加する。
```
**CookieスティーリングはHTTPOnly属性の確認が前提：**
ブラウザ DevTools の Application → Cookies タブで対象Cookieの HTTPOnly 列を確認する。
- HTTPOnly が付いている → `document.cookie` での取得は不可。DOM偽装・フィッシングリダイレクト・CSRF補助に切り替える
- HTTPOnly がない → Cookieスティーリングが有効
```

### 案件9：クラウドWebアプリ（Web_Vuln_Flow + Credential_Discovery + Reverse_Shell確認）

**指摘G｜Reverse_Shell.md のヘッダー宣言と実リンクの不一致**

`Reverse_Shell.md` の冒頭「このファイルの位置づけ」に
「`../02_Initial_Access/Web_Vulnerabilities/Command_Injection.md` から参照される」と書いてある。
しかし `Command_Injection.md` を実際に確認すると、リバースシェルペイロードは書いてあるが
`Reverse_Shell.md` へのリンクが存在しない。
`XSS.md` にはリンクが入っているが、最も直接的に使う `Command_Injection.md` から繋がっていない。

→ **修正案**：`Command_Injection.md` の Step 3（リバースシェル取得）の冒頭に
「> リバースシェルの仕組み・ポート選択・VPN環境でのIP確認 → `../../06_Concepts/Reverse_Shell.md`」
を追記する。

---

**指摘H｜Credential_Discovery.md の末尾チェックリストへの誘導がない**

「認証情報を取得したら必ず試すこと」という重要なチェックリストが末尾にまとまっている。内容は良い。
ただしパターン3（LDAP）で認証情報を取得した後にファイルを閉じると見落とす。
各パターンの注意点に「他サービスで試す」とは書いてあるが、末尾のチェックリストへの誘導リンクがない。

→ **修正案**：各パターン（1〜7）の「注意点・落とし穴」の末尾に1行追加する。
「→ 取得した認証情報の確認手順: このファイル末尾「認証情報を取得したら必ず試すこと」を参照」

---

**指摘I｜Web_Vuln_Flow.md のStep 3にBAC「認証なしアクセス」確認がない**

Step 3「認証・認可の横断確認」は「低権限ユーザーで高権限エンドポイントにアクセス可能か」と
「別ユーザーのオブジェクトIDに変えてアクセスできるか」の2パターンのみ。
Broken Access Control の基本確認として最初に行うべき「そもそも未ログイン状態でログイン後ページに直アクセスできないか」が抜けている。
Web診断2.5年目の感覚では「認証なし → 低権限横断 → 他ユーザー」の順番が自然な確認順序。

→ **修正案**：Step 3の冒頭に「確認の3層構造」を追加する。

```
認証・認可の確認は3層で行う：

1. 認証なしアクセス — 未ログイン状態で /dashboard・/admin・/api/v1/users 等を直接叩く
   → 200が返る・コンテンツが表示される → Broken Access Control（最も重大）
2. 低権限→高権限 — 一般ユーザーで管理者エンドポイントにアクセスする
3. 横断アクセス — 別ユーザーのオブジェクトID（IDOR）にアクセスする
```

---

**指摘J｜Web_Vuln_Flow.md Step 0 に業務管理的な確認事項が混入している**

「報告形式 — 脆弱性の発見だけか、PoC（概念実証）まで求められるか」という確認項目が Step 0 にある。
CLAUDE.md の「技術目線のみ。報告・スコープの内容は書かない」ルールと矛盾している。
技術的な判断に必要な情報（スコープ対象・除外エンドポイント・認証情報の有無）は残してよいが、
報告形式は技術的な手順選択に関係しない。

→ **修正案**：「報告形式」の行を削除する。「除外エンドポイント」も「本番データに触れる操作の要承認範囲」という
業務的な表現から「スコープ外のエンドポイント（管理画面・決済フロー等）」に置き換える。

## Part 2：作成者視点の修正案

### 根本的な問題 3つ（横断的な構造問題）

個別ファイルを修正しても同じ問題が別のファイルで出続ける。原因は構造にある。

---

**問題①：「攻撃者IPの指定」問題が複数ファイルに散在している**

`SSRF.md`・`XSS.md`・`Command_Injection.md` の3ファイルで「`[ATTACKER_IP]`・`[ATTACKER_HOST]`・`[自分のIP]`」という
表記が使われているが、VPN環境での `tun0` 使用の注記は `Reverse_Shell.md` にしか書かれていない。
コールバック受信系の脆弱性（SSRF・XSS格納型）でこの問題が繰り返し発生する。

これは「都度各ファイルに書く」のではなく、「コールバック受信系には Reverse_Shell.md の攻撃側準備を参照させる」
という参照構造にすることで解決できる。

→ **CLAUDE.mdへの追記**：後述の「書き方」ルール改訂案を参照。

---

**問題②：06_Concepts/ ファイルのヘッダー宣言と実リンクの乖離**

`Reverse_Shell.md` の「このファイルの位置づけ」に `Command_Injection.md` が参照元として書いてあるが、
実際のリンクが張られていない。
ファイルを作ったときに位置づけを書くが、参照元ファイルへの実際のリンク追記を忘れている、
または後から参照元が増えたときに更新されていない、というパターン。

CLAUDE.md の「06_Concepts/ の扱い」に「冒頭に参照元を書く」とあるが、
「参照元ファイルに実際のリンクを張ること」が明記されていない。

→ **CLAUDE.mdへの追記**：「06_Concepts/ の扱い」セクションに追記（後述）。

---

**問題③：「前：後：」フォーマットの遡及反映が追いついていない**

前回の修正でルールとしては追加されたが、既存ファイルへの反映が新規作成ファイルに限られている。
`XSS.md`・`SSRF.md`・`IDOR.md`・`SQLi.md`・`Command_Injection.md` がいずれも旧フォーマットのまま。

ルールを書くだけでなく「既存ファイルを編集した際に合わせて揃える」という義務をルールに追加すれば、
次回の追記タイミングで自然に修正が進む。

→ **CLAUDE.mdへの追記**：「書き方」ルール改訂案を参照。

---

### CLAUDE.md 改訂案

#### 変更1｜「書き方」ルールへの追記

以下を「書き方」セクションに追加する：

```
- コールバック受信が必要な脆弱性（Blind SSRF・格納型XSS等）には受信手段を書く
  `python3 -m http.server 8000` での受信方法と、VPN環境でのIP確認（tun0）を記載する。
  または `../06_Concepts/Reverse_Shell.md`（攻撃側の準備①②）へのリンクで代替する。

- 「何が出たら先に確認すべき条件があるか」を観点・着眼点に書く
  手順を試みる前に確認すべき前提条件がある場合（例：HTTPOnly Cookieの確認）は、
  観点・着眼点の冒頭に「先に確認すること」として分離して書く。
  注意点・落とし穴に埋めない。

- 既存ファイルを編集した際は「関連技術」を前：後：フォーマットに揃える
  新規ファイルだけでなく、既存ファイルに追記・修正を行った際も
  そのファイルの「関連技術」を「前：〜 / 後：〜」フォーマットに合わせる。
```

---

#### 変更2｜「06_Concepts/ の扱い」セクションへの追記

既存の「作業ファイルには `> 原理 → ../06_Concepts/[ファイル名].md` の1行リンクのみ追加する」
の直後に以下を追加する：

```
- 06_Concepts/ ファイルを作成した際は、冒頭「このファイルの位置づけ」に書いた
  全ての参照元ファイルに実際のリンクを張ること。
  ヘッダーに書いただけでリンクを張らないと、参照元から辿れない孤立ファイルになる。
```

### 修正の進め方（作成者メモ）

今回の指摘で明確になったことがある。**「ルールを書くこと」と「既存ファイルに反映すること」は別の作業**。
前回のREVIEW.mdの修正で CLAUDE.md のルールはきれいになったが、
既存ファイル10本以上の「関連技術」は旧フォーマットのまま残っている。

次回の追記タイミングで個別ファイルを修正する際に、
「このファイルを触ったついでに関連技術フォーマットも揃える」を習慣にするのが現実的。
一気に全ファイルを修正しようとすると追記より修正が多くなってモチベーションが下がる。

優先度「高」の4件（Command_Injection / IDOR / SSRF / Web_Vuln_Flow）だけは次の案件前に直しておきたい。
理由は、次にこれらのファイルを使った際に確実に同じ場所で詰まるとわかっているから。
