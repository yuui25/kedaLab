# ソーシャルエンジニアリング

> **[HIGH IMPACT]** 本攻撃カテゴリは本番で実施する場合に特別な事前合意が必須：
> - [x] 業務停止リスク（対象従業員の業務遂行への影響）
> - [x] 法的リスク（個人情報保護法・労働法・不正アクセス禁止法）
> - [x] 第三者影響（家族・取引先への波及可能性）
> - [x] レピュテーションリスク（顧客従業員間の信頼関係）
>
> 書面承認・対象者リスト・実施手段・通知範囲を明示的に取り決めない限り実施しないこと。
> 演習環境・社内訓練として実施する場合も「訓練」と「診断」の区分を明確にすること。

## 概要

技術的な脆弱性ではなく人間の心理を操作して、機密情報の開示・不正な操作の実行・マルウェアの実行を誘導する攻撃手法群。技術的なセキュリティ制御をバイパスして組織への足がかりを作るための初期アクセス手法として機能する。

---

## 着火条件

- 標的組織の従業員・ユーザーへの連絡手段がある（メール・電話・SNS等）
- 技術的な侵入経路がなく、または補完手段として人間を介したアクセス取得が有効な場合
- 標的の組織・業務・役割・利用ツールに関する情報が OSINT 等で入手できている

---

## 観点・着眼点

**ソーシャルエンジニアリングのシグナルチェーン：**

```
OSINT で組織情報収集（従業員名・役職・利用ツール・メールアドレス形式）
    ↓
高価値標的を選定（IT 管理者・データアクセス権保持者・管理職等）
    ↓
フィッシング / プリテキスティング / ベイティング を実行
    ↓
[認証情報を取得]                [マルウェアを実行させる]
    ↓                               ↓
直接ログイン →                  C2 接続・バックドア確立 →
技術的な侵入フェーズへ移行       技術的な侵入フェーズへ移行
```

**AI システムへの初期アクセスとしての社会工学：**

- AI システムの学習データ・推論 API・データパイプラインへの正規アクセス権を持つ従業員が主な標的になる
- 認証情報を取得すれば技術的な脆弱性なしにデータポイズニング・データ盗取・モデル操作に到達できる
- 「侵害された内部者（Compromised Insider）」はこの経路で生まれる → `06_Concepts/AI_ML/Data_Attacks.md`（インサイダー脅威のセクション）

## 手法1：フィッシング（Phishing）

信頼できる組織・個人を装ったメールや Web ページで認証情報・機密情報を詐取する手法。

| バリエーション | 特徴 |
|------------|------|
| スピアフィッシング | 特定個人・組織を標的に高度にパーソナライズ |
| ホエーリング（Whaling） | 経営幹部・管理職など高権限者を標的 |
| スミッシング | SMS を使用 |
| ビッシング | 音声通話を使用 |
| BEC（ビジネスメール詐欺） | 上司・役員を装い緊急の送金指示・情報提供を要求 |

**認証情報フィッシングの流れ：**
1. 正規サービス（VPN・メール・社内ポータル）のログインページを模した偽サイトを準備
2. 標的に偽サイトへの誘導メールを送信（緊急性・権威を装う）
3. 入力された認証情報を攻撃者サーバーに送信・収集
4. 取得した認証情報で本物のシステムに直接アクセス

**有効性が高い条件：**

- 標的の利用ツール・上司の名前・組織内のイベント等に合わせてパーソナライズされている
- 緊急性（「今すぐ確認が必要です」）や権威（「セキュリティチームより」）を強調している
- 送信元ドメインが本物に似ている（タイポスクワッティング・サブドメイン偽装）

---

## 手法2：プリテキスティング（Pretexting）

架空のシナリオを作り上げ、標的が「自発的に」情報・アクセス権を提供する状況を作り出す手法。

| 典型シナリオ | 手口 |
|-----------|------|
| IT サポートを装う | 「システム障害対応のためパスワードの確認が必要です」 |
| 監査員・コンプライアンス担当を装う | 「セキュリティ監査のためアクセス権限の確認をお願いします」 |
| ベンダー・取引先を装う | 「システム更新のため VPN 接続情報の提供が必要です」 |
| 新入社員を装う | 「アカウントのセットアップで困っているため教えていただけますか」 |

**プリテキスティングの成立条件：**

- 組織構造・利用システム・担当者名を事前に把握している（OSINT と組み合わせる）
- 権威ある立場（IT 管理者・外部監査・上司）を装っている
- 緊急性・締め切りを強調して標的の判断の余裕を奪っている

---

## 手法3：ベイティング（Baiting）

悪意あるデバイス・ファイルを意図的に放置・配布し、標的が自発的に実行するよう誘導する手法。

| 手口 | 方法 |
|------|------|
| 感染 USB ドライブの放置 | 「給与明細」「プロジェクト計画書」等のラベルを貼った USB を組織内に放置 |
| 偽ダウンロードリンク | 正規ソフトウェア・業務ツールに見せかけたマルウェアを配布 |
| QR コードの改ざん | 公共の場・印刷物の QR コードを悪意ある URL に差し替え |

---

## 注意点・落とし穴

- **技術的な侵入痕跡が少ない**：正規の認証情報を使用するためログ上は正常なアクセスに見える。行動分析（UEBA）や異常アクセス検知で初めて発覚するケースが多い
- **成功可否は事前情報の質に依存**：組織・業務・担当者に関する情報が多いほど説得力が増す → OSINT フェーズを先行させる
- **複合攻撃が基本**：フィッシングで認証情報を取得しその後技術的手法で権限昇格するパターンが主流。ソーシャルエンジニアリング単独で攻撃が完結することは少ない
- **MFA でもバイパス可能なケースがある**：リアルタイム中間者攻撃（Evilginx 等）を使ったフィッシングキットは MFA コードも透過的に中継して取得できる

## 本番での前提（必読）

### 法的・契約的留意点

ソーシャルエンジニアリングは**人を対象とする攻撃**であるため、技術調査と異なる法的フレームに服する。
書面合意なしの実施は**不正アクセス禁止法・個人情報保護法・労働法・脅迫罪等の犯罪**となりうる。

#### 実施前に書面で取り決めるべき項目

- **対象者リスト**：氏名・所属・役職・メールアドレスを書面で限定（範囲外への波及は契約違反）
- **対象手段**：メール / SMS / 電話 / 物理侵入 / SNS のうちどれを使うか
- **訓練 vs 診断の区分**：
  - 訓練：従業員教育目的、引っかかった人にフィードバックする想定
  - 診断：組織のセキュリティ態勢評価、個人を特定しない統計報告
  - 区分により事前通知・事後対応・人事評価への扱いが変わる
- **実施時間帯**：業務時間内 / 業務時間外、深夜・休日の電話禁止 等
- **通知範囲**：CISO のみ把握 / IT部門全員把握 / 全員把握なし のどれか
- **個人情報の取扱**：取得した認証情報・通話録音・撮影物の保管・破棄ポリシー（個情法対応）
- **物理侵入の事前通知**：建物管理者・警備会社・地元警察への通知の要否
- **攻撃成功時の即時報告ライン**：実認証情報取得・端末侵害発生時の連絡先と SLA
- **失敗時の挙動**：対象者がフィッシングを通報した場合に SOC への対応をどうするか

#### 実施時に守ること

- 対象者リスト外への波及を必ず防ぐ（家族・取引先のメールアドレスを誤って踏まないよう確認）
- 取得した個人情報・通話録音は暗号化保管・案件後即破棄（書面で確認を取る）
- 物理侵入時は身分証・実施合意書のコピーを携行（警察対応用）
- ベイティング（USB放置）は対象建物内のみ・回収責任を明示
- 「攻撃が止められなくなる」事態（対象者が SOC 通報して内部調査が動く等）の連鎖を想定して中断手順を共有

#### 演習環境での扱い

- HTB / OSCP 等の演習環境では本セクションの事前合意は不要
- ただし、ソーシャルエンジニアリングの「現実での慎重さ」は演習でも意識して訓練することを推奨
