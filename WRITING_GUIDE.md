# kedalab — 書き方ガイド

**書き込み作業を開始するときに Read するファイル。**
kedalab へ知見を追記・新規作成する際の書き方ルール・禁止事項・ファイルテンプレートをすべてここに集約している。

---

## kedalab の位置づけ（前提）

kedalab は「**演習環境（HTB / OSCP 等）で出会った技術を、汎用ナレッジに翻訳して蓄積する**」リポジトリ。
利用シーンとして以下を同時に想定する：

- 商用ペネトレーションテスト・脆弱性診断の現場で「この出力が出たら次は何をするか」を即引きする
- OSCP 等のペネトレ系資格演習中に「迷わず実行する」ために引く
- 演習環境（HTB 等）で技術を練習する際に判断ロジックを引く

**したがって、ファイルの中身は「特定環境に依存しない汎用ナレッジ」として書く。**
演習由来のIP・ドメイン・フラグ名・弱パスワード・ユーザー名等の固有値は持ち込まない。
学習リソースとしてHTB等への言及が必要な場合は、本体ファイルではなく `README.md` の限定された箇所にのみ書く。

**書く目的は「攻撃者の思考トレース」。**
コマンド集ではなく、「なぜそれを試すのか・何が出たら次は何をするか」の判断ロジックを残す。
コマンドだけ書いて手順書にしない。

**書く前に問う：** 「これは特定の演習を知っている人にしか伝わらない記述になっていないか？」
答えが YES なら、判断ロジックの抽出が不十分。条件・シグナル・分岐の形に書き換える。

## 公開コンテンツと作業領域の区別

新規ファイルを書く前に、配置先を以下で判定する：

| 内容 | 配置先 | 公開 |
|------|--------|------|
| 技術手順・概念（汎用化済み）| `00_Playbook/` 〜 `07_AI_Red_Teaming/` | 公開（GitHub push・kedaweb 可視）|
| 未公開 CVE 関連（embargo 中）| `_pending/` | 非公開（gitignore）|
| レビュー・メタ文書・下書き・タスク・定義 | `_workspace/<分類>/` | 非公開（gitignore）|

`_` 始まりのトップレベルディレクトリは kedalab フォルダ規約で公開対象外。
詳細: `_workspace/conventions/Folder_Convention_20260515.md`

## 守るべきルール

### 書き方

- 構成は必ずテンプレート（着火条件・観点・着眼点・手順・注意点・落とし穴・関連技術）に従う
- 汎用化して記述する。「この環境では」ではなく「この条件が揃った場合は」という形で書く
- 着火条件と手順は必ずセットで書く。コマンドだけで終わりにしない
- 失敗した手法・通用しなかった理由も記録する
- 技術目線のみ。報告・スコープ・学習ロードマップの内容は書かない

- **「何が出たら次は何をするか」のシグナルを観点・着眼点に必ず含める。**
  ツール出力の具体的な文字列・パターンと、そこから発火する次の手法を対応表や箇条書きで明示する。
  例：「`Domain:` の値が出たら hosts ファイルに登録する」「`signing required` が出たら NTLM リレーは除外する」

- ツール名が出てきたら必ず「どこで使えるか」を書く
  ペネトレ用Linuxディストリ（Kali・Parrot等）に標準搭載か、別途インストールが必要か、インターネットアクセスが必要か。
  ツールが使えない環境向けのオフライン代替手段も1行書く。

- 攻撃側（テスター端末）とターゲット側の操作を明確に区別する
  コマンドブロックが混在する場合は「# [Attacker]」「# [Target]」のコメントを付ける。
  「Kali」を端末の代名詞として使わない（特定ディストリへの依存を避ける）。

- 「刺さらなかったとき」セクションを書く
  手法が動かない場合の判断基準と次の選択肢を必ず書く。
  「〜が出たらこの手法は使えない」という条件不成立のシグナルを書く。

- システムに永続的な変更を加える操作には原状回復手順を書く
  /etc/passwd の書き換え・ユーザー作成・サービス変更など、実環境に影響が残る操作がある場合は
  注意点に「原状回復：[手順]」を必ず記載する。
  原状回復対象は案件識別子コメントマーカー方式（後述）で識別できるようにする。

- 外部リソースへの依存は注記する
  GTFOBins URL・GitHub curl・オンラインツール等、インターネットアクセスが前提の手順には
  「オフライン環境では〜を代替に使う」を書く。商用案件はインターネット遮断VLAN下が普通。

- 関連技術は双方向で書く
  「前：」と「後：」の両方を書く。

- コールバック受信が必要な脆弱性（Blind SSRF・格納型XSS等）には受信手段を書く
  受信用リスナーの起動方法と、テスター側到達可能IPの確認方法（`ip a` で全インターフェース確認）を記載する。
  または `../06_Concepts/Reverse_Shell.md`（攻撃側の準備①②）へのリンクで代替する。

- 「何が出たら先に確認すべき条件があるか」を観点・着眼点に書く
  手順を試みる前に確認すべき前提条件がある場合（例：HTTPOnly Cookieの確認、ロックアウトポリシーの取得）は、
  観点・着眼点の冒頭に「先に確認すること」として分離して書く。
  注意点・落とし穴に埋めない。

- 既存ファイルを編集した際は「関連技術」を前：後：フォーマットに揃える
  新規ファイルだけでなく、既存ファイルに追記・修正を行った際も
  そのファイルの「関連技術」を「前：〜 / 後：〜」フォーマットに合わせる。

- **00_Playbook/ に新規ファイルを作成した際は README.md の導線テーブルを更新する**
  `README.md` の「状況から直接フォルダに飛ぶ」テーブルに、新しいPlaybookファイルへの行を追加する。
  Playbookファイルは「いつ使うか」が明示されていないと存在を気づかれないため、READMEへの登録を必須とする。

- **新規エントリ追加時は該当する MITRE ATT&CK / OWASP WSTG ID も併記する**
  主インデックス（`TECHNIQUES_INDEX.md` / `TECHNIQUES_INDEX_AI_ML.md`）への追記に加えて、
  該当する ID があれば二次インデックスにも1行追記する：
  - MITRE ATT&CK Technique ID（例：`T1558.003` Kerberoasting）→ [`TECHNIQUES_INDEX_MITRE.md`](./TECHNIQUES_INDEX_MITRE.md)
  - OWASP WSTG ID（例：`WSTG-INPV-05` SQL Injection）→ [`TECHNIQUES_INDEX_WSTG.md`](./TECHNIQUES_INDEX_WSTG.md)

  **該当する ID が無い場合は無理に当てはめない**（網羅性より引きやすさ優先）。
  Web 系の手順は WSTG が、AD・Linux・ネットワーク系は ATT&CK が該当することが多い。
  AI Red Teaming は ATT&CK ではなく MITRE ATLAS のスコープ（将来 `TECHNIQUES_INDEX_ATLAS.md` を別建てで対応予定）。
  1技術が複数 ID にまたがる場合は両方の行に記載してよい（N:N で良い）。

- **Playbook ファイルには冒頭に「案件開始条件の確認」テーブルを設ける**
  「提供されている情報 → 開始位置」のマッピング表を Playbook ファイルの最初のセクションとして配置する。
  手元の情報によってスタート地点が変わるため、これがないと読み始めの方向が定まらない。
  `Linux_Attack_Flow.md`・`Windows_AD_Attack_Flow.md` の両方に存在することを維持する。

- **ツール・専門用語が当該ファイルで初登場するときは括弧内に一文の説明を付ける**
  例：`nxc`（NetExec の CLI ラッパー。SMB/WinRM/MSSQL への認証テストを一括で行う、ペネトレ用Linuxディストリに標準搭載）
  「どこで使えるか（標準搭載/要インストール）」の既存ルールと合わせて記載する。
  `05_Tools_Reference/` の詳細ファイルがある場合はリンクで誘導する。

- **コードブロックの実行前に必要な前提操作はコードコメントに埋めず独立したステップとして書く**
  環境変数の export・別ウィンドウでのリスナー起動・権限昇格後の前処理 等、
  コマンド本体より先に実行しないと失敗する操作は、コードブロックの外に「**事前準備（必須）：**」として分離する。
  コードコメント（`# 事前に〜`）に書くだけでは見落とすリスクが高い。

- **外部ファイルの出力フォーマットを参照する際は具体的な行例を1つ以上示す**
  `/etc/passwd`・`/etc/shadow`・hosts ファイル など、フォーマットが自明でないファイルの読み方を説明する場合は、
  フィールド構成の説明に加えて実際のデータを模した具体的な行例（3行程度）を必ずコードブロックで示す。
  「末尾フィールドがシェル」のような抽象説明だけで終わらせない。

- **「攻撃者の思考トレース」を書く**
  単なるコマンド列ではなく、「この状況でなぜこの手を選ぶのか」「次の出力をどう判断するのか」を残す。
  読み手が同じ思考プロセスを再現できるように、「なぜそう考えたか」を観点・着眼点に必ず1行書く。

### 禁止事項

#### 演習環境名・資格名・コース名の禁止

- 資格名・コース名・演習環境名（HTB / Hack The Box / OSCP / TryHackMe / VulnHub 等）を
  ファイル名・見出し・本文のどこにも入れない
- 試験対策・出題範囲・難易度などの資格固有の情報を書かない
- 「リタイアマシン」「ボックス」「マシン」（単一ホストを指す用法）等の演習由来の呼称を使わない
  → 「ホスト」「対象システム」「対象サーバー」と書く
- 「攻略」を使わない → 「テスト」「診断」「ペネトレ実施」と書く
- フラグファイル名（`user.txt` / `root.txt` / `flag.txt`）を例示に使わない
  → root権限取得の確認は `id` の出力、`/etc/shadow` の先頭1行、`whoami /priv` 等の客観的証跡で示す

#### 例示に使ってはいけない具体値

| 種別 | 禁止例 | 使用すべき値 |
|------|--------|-------------|
| IPアドレス | `10.10.10.10`, `10.10.14.x`, `10.10.16.x`, `10.129.x.x` | `192.0.2.x`, `198.51.100.x`, `203.0.113.x` (RFC 5737) |
| ドメイン | `corp.local`, 演習で出会った実FQDN | `example.local`, `example.test`, `example.invalid` (RFC 2606) |
| ホスト名 | `dc01`（演習想起）など | `[DC]`, `[FILESERVER]` 等のプレースホルダ、または RFC 2606 配下の名前 |
| ネットワークインターフェース | `tun0` 明示（VPN/HTB前提として書くこと） | 「テスター側の到達可能インターフェース（環境による：物理LAN・VPN・専用線等）」 |
| 弱パスワード例 | `Password123!`, `P@ssw0rd1234!` | `[STRONG_RANDOM_PASSWORD]`, `[CLIENT_PROVIDED_PASSWORD]` |
| 演習由来のアカウント名 | `henry`, `hacker`, `ATTACKPC$`, `evil.bat` | `[USER]`, `[案件略号]_TEST$`, `[script_name].bat` |
| 演習由来の固有値 | コンテナID `e6ff5b1cbc85` 等の実値 | `[CONTAINER_ID]`, `[HASH_VALUE]` 等のプレースホルダ |
| ハッシュ実値 | 演習で取得したハッシュ値そのまま | `[NTLM_HASH]`, `[KRBTGT_HASH]` |

> **`tun0` について**：VPN接続が前提のときも明示せず、「テスター側の到達可能インターフェース」と書き
> 確認手段（`ip a`）を示すに留める。「VPN」「物理LAN」「ジャンプサーバー」など特定の経路名は
> 例として括弧書きで列挙してよいが、いずれか一つを前提とした書き方をしない。

#### プレースホルダの表記規則

| 用途 | 統一表記 | 禁止表記 |
|------|---------|---------|
| 一般プレースホルダ | `[UPPER_SNAKE_CASE]`（角括弧 + 大文字スネーク） | `<lowercase>`、`{var}`、`<attacker ip>` 等 |
| 例 | `[ATTACKER_IP]`、`[TARGET_IP]`、`[USER]`、`[PASSWORD]`、`[NTLM_HASH]` | `<attacker ip>`、`{user}`、`$IP`、`%USER%` |

> 山括弧（`<...>`）は HTML / XML の例示と紛らわしく、Markdown で消えることもあるため使用しない。
> 必ず角括弧 + UPPER_SNAKE で書く。1 ファイル内・複数ファイル間で同じ意味のプレースホルダは同じ表記に揃える。

#### その他の禁止事項

- 網羅的に書こうとしない。今回扱った技術だけを対象にする
- 既存ファイルへの追記で済む場合は新しいファイルを作らない
- インデックスは必ず使い分ける
  - `TECHNIQUES_INDEX.md` → ペネトレスト・セキュリティ診断に関する技術のみ
  - `TECHNIQUES_INDEX_AI_ML.md` → AI/ML・機械学習・AI Red Teaming に関する技術のみ
  - 判断基準：「ネットワーク・システム系のペネトレ案件中に引く情報か」→ YES なら前者、NO なら後者
- AI攻撃の手順ファイルは `07_AI_Red_Teaming/` 配下に置く。`06_Concepts/AI_ML/` には「なぜその攻撃が成立するか」の原理のみ置く
- 重要な前提条件・必須操作をコードコメントのみで伝えない
  失敗の原因になる前提操作（export・事前起動・権限確認等）はコードブロックの外に書く
- **「root取得＝ゴール」と書かない。**
  権限昇格の先には「他ホストへの横展開」「保存された認証情報」「業務影響評価」が続く。
  権限昇格系ファイルの末尾には「昇格成功後の確認事項（横展開観点）」を必ず書く（後述）。

### 著作権・公開リポジトリに関するルール

- **有料学習コンテンツ由来の具体的なコード・リストはそのまま転記しない**
  コースや書籍のサンプルコードを整形しただけのものも対象。
- **以下は問題なく記載できる**
  - 概念・原理の説明（自分の言葉で書いたもの）
  - scikit-learn・pandas 等のライブラリの標準的な API 使用例
  - 公開データセット・論文・公式ドキュメント由来の情報（出典を明記する）
- **具体的なリストが必要な場合は公式ドキュメントへの参照リンクに置き換える**
- **判断基準：** そのコンテンツを持っていない第三者が読んでコース内容を実質的に再現できるレベルなら、抽象化・参照リンク化する

## 06_Concepts/ の扱い

- 「なぜその手順が効くのか」という動作原理は、作業ファイル（01〜05）には書かない
- 作業ファイルには `> 原理 → ../06_Concepts/[ファイル名].md` の1行リンクのみ追加する
- 06_Concepts/ ファイルを作成した際は、冒頭「このファイルの位置づけ」に書いた
  全ての参照元ファイルに実際のリンクを張ること。
  ヘッダーに書いただけでリンクを張らないと、参照元から辿れない孤立ファイルになる。
- 06_Concepts/ にファイルを作るのは以下の条件を満たす場合のみ：
  - 環境が変わると同じ手が使えなくなるリスクがある
  - 「なぜ」を知らないと応用が利かない・詰まったときに原因を特定できない
  - プロトコルの仕様・セキュリティ機構の動作原理など、特定の環境に依存しない普遍的な知識

**06_Concepts/ ファイルの書き方：**
- 冒頭に「このファイルの位置づけ」として、どの作業ファイルの原理を説明しているかを明記する
- 「なぜそうなるか」「どういう条件が揃うと効くか」「環境が変わったときどこを確認するか」を中心に書く
- 作業中に参照するものではないため、多少長くなっても構わない
- `関連技術` セクションは書く（kedaweb Navigator のエッジ源）が、`前：` / `後：` ラベルは不要。
  概念ファイルに手順の前後関係はないため `関連：` のみで構わない。

## Markdown 記法の前提

kedaweb（Web ビューア）のパーサーは寛容に書かれているが、書き手は以下の規範を守る。
ルールを統一しないと意図しないレイアウトの再発生・将来の AI 書き込み時の品質劣化に繋がる。

### ファイル先頭タイトルの heading

ファイル冒頭のタイトルは **H1（`# 技術名`）を使う**。kedaweb の Navigator はファイルの H1 を
セルラベルとして表示するため、H1 が無いとコードブロック内のシェルコメント（`# [Attacker] …`）が
誤マッチしてセルラベルが汚染される事故が起きる（過去に発生済み・パーサー側に H2 fallback を入れて
緩和済みだが、書き手側でも H1 を明示する）。

H2（`## 着火条件` `## 手順` 等）は**セクション見出し**として使い、タイトル用途には使わない。

```markdown
# 技術名（H1 = タイトル）

## 着火条件      （H2 = セクション）
## 観点・着眼点
## 手順
...
```

### 関連技術セクション内のパス表記

3 形式すべてパーサー側は解釈するが、書く側は次の優先順で統一する：

1. **同フォルダの兄弟ファイル** → bare 名（例: `Network_Scanning.md`）
2. **別フォルダのファイル** → `../Folder/File.md` 形式の相対パス
3. **ルート相対**（例: `01_Reconnaissance/Foo.md`）は **使わない**

bare 名は「同フォルダ」と暗黙に伝わる。ルート相対は階層感覚が掴みにくいうえ、
本文中の表など別文脈で出現するルート相対パスと混同しやすい。

### 表セル内のリテラル pipe

セル本文中で `|` を文字として書きたい場合は `\|` でエスケープする：

```markdown
| 観測される出力 | 次のアクション |
|--------------|-------------|
| `grep -i "pass\|pwd\|cred"` で抽出 | … |
```

エスケープせずに書くと、その `|` がセル区切りとして解釈され列数が崩れる。

### 太字ラベル直後にリスト/表を置くときは空行を入れる

```markdown
**シグナルと次のアクション：**

- 観測 A → 行動 X
- 観測 B → 行動 Y
```

空行を省略すると CommonMark 標準準拠のパーサー（kedaweb 含む）で
ラベルとリストが同一段落として解釈され、リストの HTML タグが文字列として表示される事故が起きる。
表（`|`）を続けるときも同様。

### 関連技術ラベルの形式バリエーション

`前：` / `後：` / `関連：` の 3 ラベルは以下のすべての書式が認識される：

```markdown
- 前：description → `Foo.md`
- 後：description → `Bar.md`
- 関連：`Baz.md`（description）
- 後（条件A）：`Foo.md`（条件付きの後続）
- 後（経路B）：`Bar.md`（複数経路を明示したい場合）
- 後（X 発見） → `Y.md`（区切り文字に `→` を使う形式）
```

**用途別の使い分け：**
- **無修飾の `前：`/`後：`**：単一の前後関係。最も一般的
- **`後（条件名）：`**：「この条件が成立した場合の後続」を明示。例: `後（GenericAll 発見）：` / `後（経路A）：`
- **`→` 区切り**：表形式や箇条書きで「ラベル + 矢印 + パス」の視覚的な流れを優先したい場合

複数バリエーションが混在しても kedaweb はすべてエッジとして拾うが、**1 ファイル内では書式を統一する**こと（可読性のため）。

### 複数の `### 関連技術` セクションを許可

長文ファイルでは「節単位の関連技術」＋「ファイル末尾の包括的な関連技術」を併用してよい。
kedaweb Navigator は全 `関連技術` セクションを集約してエッジ化する。
ただし「ファイル全体の前後関係」は **末尾の `関連技術`** に集約するのが慣習。

### コードフェンスの言語指定

```` ```bash ````、```` ```python ```` 等を必ず付ける。
shell/script 系で言語指定がないと kedaweb の syntax highlight が無効化される。

```
## 観点・着眼点
なぜこれに気づくべきか、どこを見るべきか。
「先に確認すること」「何が出たら次に何をするか」のシグナルをここに書く。
攻撃者の思考トレース：「この状況でなぜこの手を選ぶのか」を1行で残す。

## 手順
コマンド・操作。テスター側 / ターゲット側 を明示する。
混在する場合は各コマンドに「# [Attacker]」「# [Target]」のコメントを付ける。

## 攻撃側の準備（必要な場合）
攻撃者マシンで事前に起動・用意するものがある場合のみ記載する。

## 刺さらなかったとき
「〜が出たらこの手法は使えない → 代替として〜を試す」の形で書く。

## 注意点・落とし穴
失敗しやすいポイント、バージョン依存の挙動、失敗した手法の記録。
システムに変更を加える操作がある場合は必ず原状回復手順を記載する。

## 関連技術
- 前：この技術の起点になる手法
- 後：この技術の次に試すべき手法
- 関連：並列的に関連する手法（前後関係がない場合）
```

## 高影響攻撃の警告ブロック

以下の条件を一つでも満たす攻撃を扱うファイルには、技術内容の前に統一警告ブロックを置く：

- ドメイン全体・サービス全体への持続化に該当する（Golden Ticket / Silver Ticket / DCSync 自己付与等）
- 業務停止リスクがある（パスワードリセット・サービス停止を伴う設定変更・カーネルパニックリスク等）
- 不可逆な設定変更を含む（`/etc/passwd` 直接編集・`xp_cmdshell` 有効化・SPN/DNS追加等）
- SIEM/EDR で確実に検知される（DCSync・マシンアカウント作成・ASREPRoast/Kerberoast 等）

```markdown
> **[HIGH IMPACT]** 本攻撃は以下の理由で商用案件では原則禁止または個別合意必須：
> - [ ] 業務停止リスク（サービス・認証）
> - [ ] 持続化に該当
> - [ ] 不可逆な設定変更を含む
> - [ ] SIEM/EDR で確実に検知される（Event ID xxxx 等）
> 実施可否は事前合意で明示確認すること。演習環境（HTB / OSCP 等）では制約なし。
```

該当する項目にチェックを入れて使う。商用案件と演習環境で挙動が変わる場合は明示する。

## 「商用案件での前提」セクション（推奨／高影響攻撃では必須）

ファイル末尾に以下のセクションを置く。商用案件で参照されるときに必須となる観点を1か所に集約する。

```markdown
### 商用案件での前提

- **事前合意の要否**: ★★★（書面承認必須） / ★★（口頭確認可） / ★（技術的判断のみ）
- **想定されるSIEM/EDR検知**: Event ID xxxx（説明） / アラート名「...」
- **業務影響リスク**: サービス停止 / パフォーマンス低下 / なし
- **原状回復必須項目**: ✅ 作成したXX を削除 / ✅ 設定したYY を戻す / ✅ 取得した認証情報の破棄
- **取得情報の取扱**: 暗号化保管 / 案件終了時破棄 / 顧客への返却
- **演習環境での扱い**: 制約なし（HTB / OSCP 等は本セクション全項目をスキップしてよい）
```

> 「演習環境での扱い」を明記することで、同じファイルを商用と演習の両方で引ける。
> 演習中に商用前提を全部読まされて手が止まらないようにする。

## 原状回復の案件識別子コメントマーカー方式

`/etc/hosts` 改変・テスト用ユーザー作成・SPN追加など、システムに痕跡が残る操作には
案件識別子コメントマーカーを付け、原状回復時に grep で識別できるようにする。

```bash
# [Attacker] 案件識別子コメントマーカー付きで追記
echo "192.0.2.10  dc01.example.local  # kedalab-[CASE_ID]" | sudo tee -a /etc/hosts

# 原状回復
sudo sed -i.bak '/# kedalab-\[CASE_ID\]/d' /etc/hosts
```

## 権限昇格成功後の「横展開観点」セクション

権限昇格系ファイル（`03_Post_Access_Linux/*` `04_Post_Access_Windows_AD/Credential_Dumping.md` 等）の末尾には
「昇格成功後の確認事項」を定型として書く。「root 取得＝ゴール」というCTF的構造を打破する。

```markdown
### 昇格成功後に確認すること（横展開観点）

- /root/.ssh/ 配下の鍵 → 他ホストへのSSH接続性

- 内部サービスの管理画面・DB接続情報（設定ファイル / 環境変数）
```

Windows AD では SYSTEM 権限取得後の SAM/LSA dump、保存資格情報、Kerberos チケットの調査、
LAPS 値の取得などに同様のセクションを置く。

### CVE 研究・バリアントハント由来の固有値検出

CVE 研究や脆弱性報告で得た知見を kedalab に書く際は、**未公開（報告中・embargo 中）の発見** の固有値が
残っていないかを別途チェックする。演習環境固有値とは別経路の漏洩リスク。

```bash
# 検証中・未公開のプロジェクト名・組織名（追加で検出したいものは随時追記）
grep -nE "TriliumNext|trilium|siyuan|Joplin|Logseq|StandardNotes|Notable" [編集したファイル]

# 起点 CVE のソースコード由来の関数名・ファイル名・行番号
grep -nE "escapeAriaLabel|revisions\.ts|popover\.ts|tooltip\.ts|escape\.ts" [編集したファイル]
grep -nE ":\s*[0-9]{2,4}\s*$|\.ts:[0-9]+|\.js:[0-9]+" [編集したファイル]

# 製品バージョン pinning（v0.95.0 のような具体値は未公開発見の対象バージョンを暴露する）
grep -nE "v[0-9]+\.[0-9]+\.[0-9]+" [編集したファイル]

# 未公開 CVE 番号 / 直近年の CVE は要注意（公開済みなら明示してよい）
grep -nE "CVE-202[5-9]-[0-9]+" [編集したファイル]
# ヒットした場合: NVD で当該 CVE が published 状態か確認。Reserved / Rejected のまま記載しない。

# 「実例」「ケーススタディ」セクションの存在自体を要確認
grep -nE "^##\s*実例|^##\s*ケーススタディ|^##\s*事例" [編集したファイル]
# ヒットした場合: そのセクションに固有値が残っていないかを目視確認。
# kedalab は「演習・案件由来の知見を汎用化する場」なので、
# 実例は必ず [CANDIDATE_REPO] / [CLIENT_RENDERING_FILE] 等のプレースホルダで書く。
```

**chunk テスト：** 上記 grep がヒットしたら、ヒットした行を以下のいずれかで処理する：
1. 公開済み CVE への参照なら維持してよい（NVD で published を確認した上で）
2. 未公開 / 報告中の発見由来なら **角括弧プレースホルダに戻す**
3. 起点 CVE の説明として残すなら **「バグクラス」表現（条件 + sink + 結果）** に書き換える

### 相対パスリンクの深さ誤り検出

`02_Initial_Access/Web_Vulnerabilities/` のように **2 階層下** にあるファイルから kedalab ルートへ戻るには `../../` で十分。`../../../` は kedalab の**外**を指してしまうのでリンク切れになる：

### Markdown 記法のチェック

```bash
# F1: 関連技術内でルート相対パスを使っていないか
#     `01_Reconnaissance/...`・`02_Initial_Access/...` 等を関連技術節内で見つけたらNG
awk '/^#{2,6} 関連技術/{inrel=1} /^#{1,6} /{if($0 !~ /^#{2,6} 関連技術/) inrel=0} inrel && /`0[0-8]_[A-Za-z_]+\/[^`]+\.md`/' [編集したファイル]

# F2: 太字ラベル直後の空行欠落（リスト/表が壊れる原因）
grep -nPzo '\*\*[^\n]+[:：]\*\*\n[-*|]' [編集したファイル]

# F3: 関連技術内のディレクトリ参照（末尾 `/` 終わり・kedaweb はリンクとして拾えない）
awk '/^#{2,6} 関連技術/{inrel=1} /^#{1,6} /{if($0 !~ /^#{2,6} 関連技術/) inrel=0} inrel && /`[^`]+\/`/' [編集したファイル]
```

ヒットしたら：
- F1 → `./` または `../` で始まる相対パスに書き換える
- F2 → ラベルとリスト/表の間に空行を 1 行入れる
- F3 → 末尾 `/` のディレクトリ参照を具体的な `*.md` ファイル名に置き換える（複数ある場合は行を分割）
