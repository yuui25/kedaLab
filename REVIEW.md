# kedalab レビューメモ

## このファイルについて

kedalab 改善のための記録。  
以下の2つの視点からまとめている。

- **診断員視点**：Web脆弱性診断2.5年目・ペネトレスト初経験者が仮想的に6案件でkedalabを使用した際の指摘（計23件）
- **作成者視点**：指摘を踏まえたREADMEのAI指示の修正案・構造改善案

## Part 1：診断員視点の指摘

### 案件1：ECサイト Linux/Webアプリ（初案件）

**指摘1｜READMEの「使い方」が熟練者前提になっている**

「環境が判明したら、まず `00_Playbook/` の該当フローを開く」は「環境が判明した後」の話しか書いていない。
「調査の大きな流れ（偵察→侵入→昇格）が何段階あるのか」の概観がフォルダ構成の説明の前にない。
IPだけ渡された最初の一手として何をするかの前提が抜けている。

→ **修正案**：フォルダ構成の前に「攻撃フェーズの全体像」を1段落追加する。

---

**指摘2｜`Linux_Attack_Flow.md` Step 2、gobusterだけリンクがない**

Step 2の他の項目（robots.txt・JSソース・コマンドインジェクション）にはリンクがあるのに、
「gobuster / ffuf でディレクトリ列挙」の行だけリンクが段落末尾の汎用リンクに紛れていて見落とす。
さらに `05_Tools_Reference/` にgobusterのページがない。

→ **修正案**：gobusterの行に `Web_Enumeration.md` の直接リンクを追加する。

---

**指摘3｜リバースシェルの概念説明がゼロ**

`Linux_Attack_Flow.md` Step 4 経路B に `nc -lvnp 4444` が突然登場する。
「なぜ攻撃側がリスナーになって標的から接続させるのか」という仕組みの説明がない。
`Command_Injection.md` に飛んでも概念説明はなくコマンド例が並ぶだけ。

→ **修正案**：`06_Concepts/Reverse_Shell.md` を作り、概念を説明する。各ファイルから1行リンクを張る。

---

**指摘4｜シェル安定化の `stty raw -echo` 手順の途中が説明不足（→解決済み）**

`Shell_Stabilization.md` を確認すると「stty raw -echo の意味」の表があり、各部分の説明が入っている。
ただし `stty raw -echo` 実行後の「画面が無応答に見えるがそのまま fg と打つ」という部分が
注意点にはあるが手順本文に目立つ形で入っていない。初見でCtrl+Z後に何が起きているか不安になる。

→ **修正案**：手順 Step 3〜5 に「何が起きているか」の1行コメントを追加する。

### 案件2：社内AD環境（2案件目）

**指摘5｜ASREPRoastingの「ユーザーリストがあれば」が宙に浮いている**

`Windows_AD_Attack_Flow.md` Step 2 で「ユーザーリストがあれば試す」とあるが、

**指摘6｜ldapsearchの `-b "DC=[domain],DC=[tld]"` に具体例がない**

nmap 出力の `corp.local` が ldapsearch の `-b "DC=corp,DC=local"` に対応することが初見にはわからない。

→ **修正案**：ldapsearch コマンドの直後に「例：ドメインが `corp.local` なら `-b "DC=corp,DC=local"`」を追記する。

### 案件3：クラウドSaaS（Webのみスコープ）

フロー形式の案内がない。

→ **修正案**：`00_Playbook/Web_Vuln_Flow.md` を追加する（Webのみスコープ向けフロー）。

### 案件4：監視ツール（Grafana）CVEから侵入してLinux権限昇格

### 案件5：SQLi侵入 → Python改造 → カーネルエクスプロイト

**指摘18｜`gobuster vhost` の `--append-domain` がバージョン依存なのに注意なし**

gobuster v3.2以降の新オプションで、古いバージョンでは `unknown flag: --append-domain` で落ちる。
バージョン依存の注意がなく、コマンドの書き方を間違えたと思って長時間ハマった。

→ **修正案**：注意点に「v3.2未満では `--append-domain` が使えない。`gobuster --version` で確認するか `ffuf` を代替にする」を追記する。

**指摘19｜LinPEASのcurlがターゲットにインターネット接続がないと詰む**

一方 `Kernel_Exploits.md` に「python3 -m http.server でKaliから配る」パターンが書いてあるが、
Enumeration_Checklist.md からは見えない。

→ **修正案**：LinPEAS の行の直後に「ターゲットがインターネットに出られない場合 → Kali側でHTTPサーバーを立ててwgetで取得する（`Kernel_Exploits.md` のファイル転送パターンを参照）」を追記する。

### 案件6：Windows AD → GPP → BloodHound → ACE abuse → DCSync

**指摘20｜`secretsdump.py -k` でKerberosチケットの準備手順が抜けている**

`Windows_AD_Attack_Flow.md` Step 7 の `-k` オプションはKerberosチケットを使う意味だが、
`KRB5CCNAME` 環境変数の設定手順が書いていない。
そのままコピーして実行すると `KDC_ERR_PREAUTH_FAILED` が出て原因特定に時間がかかる。

→ **修正案**：コマンドの直前に「事前に `export KRB5CCNAME=[チケットファイルパス]` を実行すること」を追記する。

**指摘21｜`netexec` が突然登場するが説明も参照先もない**

`Credential_Discovery.md` に `netexec smb` が出てくるが、
`05_Tools_Reference/` にページがなくTECHNIQUES_INDEXにも記載がない。
CrackMapExec の後継であること・インストール方法・古いKaliでは crackmapexec になることが書いていない。

→ **修正案**：`05_Tools_Reference/Netexec.md` を追加する。TECHNIQUES_INDEXにも追記する。

---

**指摘22｜`Enumeration_Checklist.md` のグループ表、`video` と `kmem` だけリンクなし**

グループ一覧表で `staff`・`docker`・`disk`・`shadow`・`adm` には対応する手法やリンクがあるのに、
`video`（スクリーンショット取得）と `kmem`（カーネルメモリ読み取り）だけ記述が中途半端で終わっている。

→ **修正案**：対応する手法が未記載なら「※手法未記載」と明示するか、エントリを削除する。

---

**指摘23｜PAMハイジャックの「SSH再ログインが引き金」、www-data では使えない**

`Linux_Attack_Flow.md` Step 6 の4条件目「SSH ログインを引き金にできる」は、
www-data でWebシェルを取った場合には満たせない。
条件が満たせない場合の代替手段がなく、staffグループに入っているのに詰まるパターンになる。

→ **修正案**：「この条件が満たせない場合（www-data等）はこの手法は使えない。別の昇格経路を探す」を明示する。

## Part 2：作成者視点の修正案

### 根本的な問題3つ

個別ファイルを修正しても同じ問題が別のファイルで出続ける。原因は構造にある。

**問題①：テンプレートに「書くべき項目」が足りない**

現行テンプレートは「手法を記録する」には十分だが「初めての人間が使う」には以下が足りない。
- ツール前提（何がどのマシンに入っている必要があるか）
- Kali側 vs ターゲット側の区別
- 刺さらなかったときの次の手
- システムへの影響と原状回復

テンプレートに定義されていないから、誰が書いても同じ抜けが生まれる。

**問題②：「関連技術」が常に前向きリンクのみ**

「どこから来たか」がわかるリンクがないため、詰まったときにPlaybookまで戻るしかない。

**問題③：TECHNIQUES_INDEXの用途が2つに割れている**

ペネトレ作業中に素早く引くためのインデックスと、AI/ML概念ノートのインデックスが同居している。
使い方が根本的に異なる情報を同じファイルに置いている。

### READMEのAI指示：変更箇所

#### 変更1｜テンプレートの改訂

```
## [技術名]

### 着火条件
この技術を試すべき状況・前提条件

### 環境前提
- 実行環境: Kali / ターゲット / どちらでも
- 必要なツール: [ツール名]（Kali標準 / 別途インストール要 / インターネットアクセス要）
- 外部リソース依存がある場合はオフライン代替手段も記載する

### 観点・着眼点
なぜこれに気づくべきか、どこを見るべきか。
「何が出たら次に何をするか」のシグナルをここに書く。

### 手順
コマンド・操作。
Kali側 / ターゲット側 を明示する。
混在する場合は各コマンドに「# [Kali]」「# [Target]」のコメントを付ける。

### 攻撃側の準備（必要な場合）
攻撃者マシンで事前に起動・用意するものがある場合のみ記載する。
例：ncリスナー起動、HTTPサーバー起動、受信サーバーのIPアドレス確認

### 刺さらなかったとき
条件が揃っていても手法が動作しない場合の確認点と次の選択肢。
「〜が出たらこの手法は使えない → 代替として〜を試す」の形で書く。

### 注意点・落とし穴
失敗しやすいポイント、バージョン依存の挙動、失敗した手法の記録。
システムに変更を加える操作がある場合は必ず原状回復手順を記載する。

### 関連技術
- 前：この技術の起点になる手法（「どこから来たか」）
- 後：この技術の次に試すべき手法（「どこへ行くか」）
```

#### 変更2｜「守るべきルール」への追記

【書き方】に以下を追加する：

```
- ツール名が出てきたら必ず「どこで使えるか」を書く
  Kali標準搭載か、別途インストールが必要か、インターネットアクセスが必要か。
  ツールが使えない環境向けのオフライン代替手段も1行書く。

- 攻撃側（Kali）とターゲット側の操作を明確に区別する
  コマンドブロックが混在する場合は「# [Kali]」「# [Target]」のコメントを付ける。

- 「刺さらなかったとき」セクションを書く
  手法が動かない場合の判断基準と次の選択肢を必ず書く。
  「〜が出たらこの手法は使えない」という条件不成立のシグナルを書く。

- システムに永続的な変更を加える操作には原状回復手順を書く
  /etc/passwd 書き換え・ユーザー作成・サービス変更など、
  実環境に影響が残る操作がある場合は注意点に「原状回復：[手順]」を必ず記載する。

- 外部リソースへの依存は注記する
  GTFOBins URL・GitHub curl・オンラインツール等、インターネットアクセスが前提の
  手順には「オフライン環境では〜を代替に使う」を書く。

- 関連技術は双方向で書く
  「前：」と「後：」の両方を書く。
```

#### 変更4｜タスク手順（手順①〜⑤）の改訂

```
① 今回使った・学んだ技術を洗い出す

② TECHNIQUES_INDEX.md を開き、既に記載されているか確認

   ・記載なし → 新しいエントリを追加し、.md に内容を書く

③ ファイルを書く際は以下を確認する：
   a. ツール名が出てきたら「どこで使えるか」を書いたか

   e. 「関連技術」に「前：」と「後：」の両方を書いたか

④ 00_Playbook/ の該当フローに分岐点として追加できるか確認する
   ・「刺さらなかった場合の代替分岐」も追加できないか確認する

⑤ 関連する他の .md の「関連技術」の「後：」に今回のファイルへのリンクを追記する
   （双方向リンクの維持）

⑥ 「なぜ効いたか」が自分でうまく説明できない場合は 06_Concepts/ に原理をメモする
```
